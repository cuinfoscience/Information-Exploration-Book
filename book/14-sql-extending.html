14  Extending Data with SQL

Loading and cleaning produce reliable tables. Extending transforms those tables into answers. This chapter introduces the SQL tools for computing metrics and testing analytical claims: GROUP BY with aggregate functions, subqueries that embed one query inside another, and Common Table Expressions (CTEs) that name intermediate results for clarity and reuse. Together, these tools let you write complex analytical pipelines entirely in SQL — pipelines that are readable, auditable, and reproducible.

The dataset for this chapter is the Boulder County Assessor’s buildings file, which contains one row per improvement section recorded for each parcel. It is a rich, messy, real-world dataset with dozens of columns describing building characteristics — year built, square footage, quality grade, heating system, roof type, and more.

14.1 Setup

import sqlite3
import pandas as pd
import numpy as np

pd.set_option('display.max_columns', 20)
pd.set_option('display.max_colwidth', 55)
pd.set_option('display.float_format', '{:.2f}'.format)
# Load the CSV, clean, and write to a local SQLite database
buildings_raw = pd.read_csv('buildings.csv')

# Strip trailing whitespace from text columns
for col in buildings_raw.select_dtypes(include='object').columns:
    buildings_raw[col] = buildings_raw[col].str.strip()

# Write to database
conn = sqlite3.connect('boulder_buildings.db')
buildings_raw.to_sql('buildings', conn, if_exists='replace', index=False)

print(f"Loaded {len(buildings_raw):,} rows into buildings table")

14.2 Part 1: GROUP BY and Aggregate Functions

14.2.1 What GROUP BY does

GROUP BY collapses rows that share the same value(s) in one or more columns into a single summary row. It is the SQL equivalent of pandas’ groupby().agg() and the conceptual equivalent of a spreadsheet pivot table.

The key rule: every column in your SELECT must be either listed in GROUP BY or wrapped in an aggregate function. You cannot select a raw column alongside grouped aggregates — SQL would not know which row’s value to display.

pd.read_sql_query("""
    SELECT
        quality_grade,
        COUNT(*) AS n_buildings,
        AVG(above_ground_sqft) AS avg_sqft,
        MIN(year_built) AS oldest,
        MAX(year_built) AS newest
    FROM buildings
    WHERE quality_grade IS NOT NULL
    GROUP BY quality_grade
    ORDER BY n_buildings DESC;
""", conn)

14.2.2 SQLite aggregate functions

Function Purpose Notes
COUNT(*) Number of rows in group Counts all rows, including NULLs
COUNT(col) Number of non-NULL values Skips NULLs — use to measure completeness
SUM(col) Sum of values Returns NULL if all values are NULL
TOTAL(col) Sum of values Returns 0.0 if all values are NULL (safer)
AVG(col) Mean of non-NULL values Ignores NULLs — denominator is count of non-NULL
MIN(col) / MAX(col) Extremes Works on text (alphabetical) and numbers
GROUP_CONCAT(col) Concatenate values into a string Useful for inventory lists

The distinction between COUNT(*) and COUNT(col) is critical. COUNT(*) counts all rows regardless of NULL values. COUNT(col) counts only rows where col is not NULL. Comparing the two tells you the completeness rate of a column.

pd.read_sql_query("""
    SELECT
        COUNT(*) AS total_rows,            -- all rows, including NULLs
        COUNT(year_built) AS non_null_year, -- only rows where year_built is not NULL
        COUNT(*) - COUNT(year_built) AS null_year,  -- the difference = number of NULLs
        -- 100.0 forces floating-point division; integer / integer would round to 0 or 1
        ROUND(100.0 * COUNT(year_built) / COUNT(*), 1) AS pct_complete
    FROM buildings;
""", conn)

14.2.3 HAVING vs. WHERE

Both filter rows, but they operate at different stages of query execution. WHERE filters individual rows before grouping. HAVING filters groups after aggregation.

# WHERE: filter rows before grouping
pd.read_sql_query("""
    SELECT quality_grade, COUNT(*) AS n
    FROM buildings
    WHERE year_built >= 2000
    GROUP BY quality_grade
    ORDER BY n DESC;
""", conn)
# HAVING: filter groups after aggregation
pd.read_sql_query("""
    SELECT quality_grade, COUNT(*) AS n
    FROM buildings
    GROUP BY quality_grade
    HAVING n >= 100
    ORDER BY n DESC;
""", conn)

A useful mnemonic: WHERE answers “which rows go into the groups?” while HAVING answers “which groups appear in the output?”

14.2.4 CASE WHEN inside aggregation

CASE WHEN expressions inside aggregate functions enable conditional counting — the SQL equivalent of COUNTIF in a spreadsheet. This is one of the most powerful patterns in SQL.

# CASE WHEN inside SUM() is the SQL equivalent of COUNTIF in a spreadsheet:
# the CASE expression returns 1 when the condition is true, 0 otherwise.
# Summing these 1s and 0s counts how many rows satisfy each condition.
pd.read_sql_query("""
    SELECT
        (year_built / 10) * 10 AS decade,  -- integer division truncates, e.g. 1987/10=198, *10=1980
        SUM(CASE WHEN bldg_class = 'R' THEN 1 ELSE 0 END) AS residential,
        SUM(CASE WHEN bldg_class = 'C' THEN 1 ELSE 0 END) AS commercial,
        SUM(CASE WHEN bldg_class NOT IN ('R', 'C') THEN 1 ELSE 0 END) AS other,
        COUNT(*) AS total
    FROM buildings
    WHERE year_built IS NOT NULL
    GROUP BY decade
    ORDER BY decade;
""", conn)

14.2.5 Working with Dates in SQL: strftime()

Government datasets often store dates as separate integer columns rather than a single date column. The cannabis sales data stores month and year as separate integers. SQL can reconstruct a date string from these using strftime() and string concatenation.

# First, load the cleaned cannabis data if you have it
cannabis_conn = sqlite3.connect('boulder_buildings.db')  # reuse same connection

pd.read_sql_query("""
    SELECT
        strftime('%Y-Q', year || '-' || printf('%02d', month) || '-01') AS quarter,
        SUM(retail_sales) AS total_retail,
        COUNT(DISTINCT county) AS n_counties
    FROM cannabis_sales
    WHERE county != 'Total'
    GROUP BY quarter
    ORDER BY quarter
    LIMIT 12;
""", conn)

printf('%02d', month) left-pads the month integer with a zero so that 3 becomes 03. SQLite’s strftime() then accepts the reconstructed date string 'YYYY-MM-DD' and formats it however you need.

TipSQLite date functions are string-based

Unlike PostgreSQL or MySQL, SQLite has no native DATE type. Dates are stored as text, integer, or real values, and strftime() operates on text in ISO format (YYYY-MM-DD). As long as your date strings are in that format, strftime() gives you full access to year, month, quarter, day-of-week, and Julian day calculations.

14.4 Part 2: Subqueries

A subquery is a SELECT statement nested inside another query. It lets you compute a value or a set of values dynamically and use the result in a filter, comparison, or derived table — without hardcoding anything.

14.4.1 Scalar subquery: comparing each row to a reference value

A scalar subquery returns a single value. You can use it anywhere a literal value would go:

# Find buildings larger than the overall average
pd.read_sql_query("""
    SELECT strap, year_built, above_ground_sqft
    FROM buildings
    WHERE above_ground_sqft > (SELECT AVG(above_ground_sqft) FROM buildings)
    ORDER BY above_ground_sqft DESC
    LIMIT 10;
""", conn)

The subquery (SELECT AVG(above_ground_sqft) FROM buildings) computes the mean once, and the outer query uses it to filter rows. This is more robust than hardcoding the average — if the data changes, the query automatically adapts.

14.4.2 WHERE IN: filtering against a dynamically computed set

# Find buildings in quality grades that have above-average mean square footage
pd.read_sql_query("""
    SELECT strap, quality_grade, above_ground_sqft
    FROM buildings
    WHERE quality_grade IN (
        SELECT quality_grade
        FROM buildings
        WHERE quality_grade IS NOT NULL
        GROUP BY quality_grade
        HAVING AVG(above_ground_sqft) > (SELECT AVG(above_ground_sqft) FROM buildings)
    )
    ORDER BY above_ground_sqft DESC
    LIMIT 10;
""", conn)

14.4.3 Derived table: wrapping aggregate results

A subquery in the FROM clause creates a temporary table (called a derived table) that the outer query can filter, join, or aggregate further:

# Rank quality grades by their median-like percentile
pd.read_sql_query("""
    SELECT *
    FROM (
        SELECT
            quality_grade,
            COUNT(*) AS n,
            ROUND(AVG(above_ground_sqft), 0) AS avg_sqft
        FROM buildings
        WHERE quality_grade IS NOT NULL
        GROUP BY quality_grade
    ) AS grade_stats
    WHERE n >= 50
    ORDER BY avg_sqft DESC;
""", conn)

14.6 Part 3: Common Table Expressions (CTEs)

A CTE is a named temporary result set defined at the top of a query with WITH ... AS. It makes complex queries readable by breaking them into named, sequential steps — the SQL equivalent of assigning intermediate DataFrames to variables in pandas.

The key distinction between a CTE and a subquery is organizational, not functional: both compute the same result, but a CTE gives the intermediate result a name. Use a subquery when the computation is simple and used once. Use a CTE when the intermediate result is complex, used more than once, or when naming it would make the overall query easier to read and audit.

14.6.1 The WITH … AS pattern

Compare these two approaches to the same question — “which quality grades have above-average square footage?”:

-- Subquery style (hard to read at depth)
SELECT * FROM (
    SELECT quality_grade, AVG(above_ground_sqft) AS avg_sqft
    FROM buildings GROUP BY quality_grade
) WHERE avg_sqft > (SELECT AVG(above_ground_sqft) FROM buildings);

-- CTE style (name the intermediate result)
WITH grade_stats AS (
    SELECT quality_grade, AVG(above_ground_sqft) AS avg_sqft
    FROM buildings
    GROUP BY quality_grade
)
SELECT *
FROM grade_stats
WHERE avg_sqft > (SELECT AVG(above_ground_sqft) FROM buildings);

Both produce the same result. The CTE version is easier to read, debug, and extend.

14.6.2 Single CTE

pd.read_sql_query("""
    WITH grade_stats AS (
        SELECT
            quality_grade,
            COUNT(*) AS n,
            ROUND(AVG(above_ground_sqft), 0) AS avg_sqft,
            ROUND(AVG(year_built), 0) AS avg_year
        FROM buildings
        WHERE quality_grade IS NOT NULL
        GROUP BY quality_grade
    )
    SELECT *
    FROM grade_stats
    WHERE n >= 50
    ORDER BY avg_sqft DESC;
""", conn)

14.6.3 Chained CTEs: multi-step analytical pipeline

You can define multiple CTEs separated by commas, each referencing the ones defined before it:

pd.read_sql_query("""
    WITH era_stats AS (
        SELECT
            (year_built / 10) * 10 AS decade,
            COUNT(*) AS n,
            ROUND(AVG(above_ground_sqft), 0) AS avg_sqft
        FROM buildings
        WHERE year_built IS NOT NULL
        GROUP BY decade
    ),
    overall AS (
        SELECT ROUND(AVG(above_ground_sqft), 0) AS grand_avg
        FROM buildings
        WHERE year_built IS NOT NULL
    )
    SELECT
        e.decade,
        e.n,
        e.avg_sqft,
        o.grand_avg,
        ROUND(e.avg_sqft - o.grand_avg, 0) AS diff_from_avg
    FROM era_stats e, overall o
    ORDER BY e.decade;
""", conn)

This pattern — define intermediate results as named CTEs, then combine them in a final SELECT — is the standard way to build complex SQL analyses. It makes every step visible and auditable.

14.9 Quick Reference

14.9.1 Subquery patterns

  • Scalar: returns one value → use in WHERE, SELECT, or HAVING
  • Column: returns one column → use with IN or NOT IN
  • Derived table: returns a table → use in FROM clause

14.9.2 CTE pattern

WITH step1 AS (
    SELECT ... FROM ... GROUP BY ...
),
step2 AS (
    SELECT ... FROM step1 WHERE ...
)
SELECT ... FROM step2;

CTEs are evaluated once, named for clarity, and discarded after the query completes. Use them whenever a query needs more than one level of aggregation or filtering.

## Part 1: GROUP BY and Aggregate Functions

### What GROUP BY does

`GROUP BY` collapses rows that share the same value(s) in one or more columns into a single summary row. It is the SQL equivalent of pandas' `groupby().agg()` and the conceptual equivalent of a spreadsheet pivot table.

The key rule: every column in your `SELECT` must be either listed in `GROUP BY` or wrapped in an aggregate function. You cannot select a raw column alongside grouped aggregates — SQL would not know which row's value to display.

```{python}
pd.read_sql_query("""

        quality_grade,
        COUNT(*) AS n_buildings,

        MIN(year_built) AS oldest,

    FROM buildings

```

### SQLite aggregate functions

| Function | Purpose | Notes |
|----------|---------|-------|
| `COUNT(*)` | Number of rows in group | Counts all rows, including NULLs |
| `COUNT(col)` | Number of non-NULL values | Skips NULLs — use to measure completeness |
| `SUM(col)` | Sum of values | Returns NULL if all values are NULL |
| `TOTAL(col)` | Sum of values | Returns 0.0 if all values are NULL (safer) |

pd.read_sql_query("""

### HAVING vs. WHERE

Both filter rows, but they operate at different stages of query execution. `WHERE` filters individual rows *before* grouping. `HAVING` filters groups *after* aggregation.

```{python}
# WHERE: filter rows before grouping
pd.read_sql_query("""
    SELECT quality_grade, COUNT(*) AS n
    FROM buildings
    WHERE year_built >= 2000
    GROUP BY quality_grade
    ORDER BY n DESC;
""", conn)
```

```{python}
# HAVING: filter groups after aggregation
pd.read_sql_query("""
    SELECT quality_grade, COUNT(*) AS n
    FROM buildings
    GROUP BY quality_grade
    HAVING n >= 100
    ORDER BY n DESC;
""", conn)
```

A useful mnemonic: `WHERE` answers "which rows go into the groups?" while `HAVING` answers "which groups appear in the output?"

### CASE WHEN inside aggregation

`CASE WHEN` expressions inside aggregate functions enable conditional counting — the SQL equivalent of `COUNTIF` in a spreadsheet. This is one of the most powerful patterns in SQL.

```{python}

pd.read_sql_query("""

```

## Part 2: Subqueries

A subquery is a `SELECT` statement nested inside another query. It lets you compute a value or a set of values dynamically and use the result in a filter, comparison, or derived table — without hardcoding anything.

### Scalar subquery: comparing each row to a reference value

A scalar subquery returns a single value. You can use it anywhere a literal value would go:

```{python}
# Find buildings larger than the overall average
pd.read_sql_query("""
    SELECT strap, year_built, above_ground_sqft
    FROM buildings
    WHERE above_ground_sqft > (SELECT AVG(above_ground_sqft) FROM buildings)
    ORDER BY above_ground_sqft DESC
    LIMIT 10;
""", conn)
```

The subquery `(SELECT AVG(above_ground_sqft) FROM buildings)` computes the mean once, and the outer query uses it to filter rows. This is more robust than hardcoding the average — if the data changes, the query automatically adapts.

### WHERE IN: filtering against a dynamically computed set

```{python}
# Find buildings in quality grades that have above-average mean square footage
pd.read_sql_query("""
    SELECT strap, quality_grade, above_ground_sqft
    FROM buildings
    WHERE quality_grade IN (
        SELECT quality_grade
        FROM buildings
        WHERE quality_grade IS NOT NULL
        GROUP BY quality_grade
        HAVING AVG(above_ground_sqft) > (SELECT AVG(above_ground_sqft) FROM buildings)
    )
    ORDER BY above_ground_sqft DESC
    LIMIT 10;
""", conn)
```

### Derived table: wrapping aggregate results

A subquery in the `FROM` clause creates a temporary table (called a derived table) that the outer query can filter, join, or aggregate further:

```{python}
# Rank quality grades by their median-like percentile
pd.read_sql_query("""
    SELECT *
    FROM (
        SELECT
            quality_grade,
            COUNT(*) AS n,
            ROUND(AVG(above_ground_sqft), 0) AS avg_sqft
        FROM buildings
        WHERE quality_grade IS NOT NULL
        GROUP BY quality_grade
    ) AS grade_stats
    WHERE n >= 50
    ORDER BY avg_sqft DESC;
""", conn)
```

## Part 3: Common Table Expressions (CTEs)

A CTE is a named temporary result set defined at the top of a query with `WITH ... AS`. It makes complex queries readable by breaking them into named, sequential steps — the SQL equivalent of assigning intermediate DataFrames to variables in pandas.

The key distinction between a CTE and a subquery is organizational, not functional: both compute the same result, but a CTE gives the intermediate result a name. Use a subquery when the computation is simple and used once. Use a CTE when the intermediate result is complex, used more than once, or when naming it would make the overall query easier to read and audit.

### The WITH ... AS pattern

Compare these two approaches to the same question — "which quality grades have above-average square footage?":

```sql
-- Subquery style (hard to read at depth)
SELECT * FROM (
    SELECT quality_grade, AVG(above_ground_sqft) AS avg_sqft
    FROM buildings GROUP BY quality_grade
) WHERE avg_sqft > (SELECT AVG(above_ground_sqft) FROM buildings);

-- CTE style (name the intermediate result)
WITH grade_stats AS (
    SELECT quality_grade, AVG(above_ground_sqft) AS avg_sqft
    FROM buildings
    GROUP BY quality_grade
)
SELECT *
FROM grade_stats
WHERE avg_sqft > (SELECT AVG(above_ground_sqft) FROM buildings);
```

Both produce the same result. The CTE version is easier to read, debug, and extend.

### Single CTE

```{python}
pd.read_sql_query("""
    WITH grade_stats AS (
        SELECT
            quality_grade,
            COUNT(*) AS n,
            ROUND(AVG(above_ground_sqft), 0) AS avg_sqft,
            ROUND(AVG(year_built), 0) AS avg_year
        FROM buildings
        WHERE quality_grade IS NOT NULL
        GROUP BY quality_grade
    )
    SELECT *
    FROM grade_stats
    WHERE n >= 50
    ORDER BY avg_sqft DESC;
""", conn)
```

### Chained CTEs: multi-step analytical pipeline

You can define multiple CTEs separated by commas, each referencing the ones defined before it:

```{python}
pd.read_sql_query("""
    WITH era_stats AS (
        SELECT
            (year_built / 10) * 10 AS decade,
            COUNT(*) AS n,
            ROUND(AVG(above_ground_sqft), 0) AS avg_sqft
        FROM buildings
        WHERE year_built IS NOT NULL
        GROUP BY decade
    ),
    overall AS (
        SELECT ROUND(AVG(above_ground_sqft), 0) AS grand_avg
        FROM buildings
        WHERE year_built IS NOT NULL
    )
    SELECT
        e.decade,
        e.n,
        e.avg_sqft,
        o.grand_avg,
        ROUND(e.avg_sqft - o.grand_avg, 0) AS diff_from_avg
    FROM era_stats e, overall o
    ORDER BY e.decade;
""", conn)
```

This pattern — define intermediate results as named CTEs, then combine them in a final `SELECT` — is the standard way to build complex SQL analyses. It makes every step visible and auditable.

## SQL ↔ pandas Equivalents

| SQL | pandas |
|-----|--------|
| `GROUP BY col` | `df.groupby('col')` |
| `COUNT(*)` | `.size()` or `.count()` |
| `SUM(col)`, `AVG(col)` | `.agg({'col': 'sum'})`, `.agg({'col': 'mean'})` |
| `HAVING COUNT(*) > 10` | Filter after `.groupby().agg()` |
| `CASE WHEN ... END` | `np.where()` or `pd.cut()` |
| Scalar subquery | Compute value, then use in filter |
| CTE (`WITH ... AS`) | Assign intermediate DataFrame to a variable |
| `WHERE col IN (subquery)` | `df[df['col'].isin(other_df['col'])]` |

## Quick Reference

### Subquery patterns

- **Scalar**: returns one value → use in `WHERE`, `SELECT`, or `HAVING`
- **Column**: returns one column → use with `IN` or `NOT IN`
- **Derived table**: returns a table → use in `FROM` clause

### CTE pattern

```sql
WITH step1 AS (
    SELECT ... FROM ... GROUP BY ...
),
step2 AS (
    SELECT ... FROM step1 WHERE ...
)
SELECT ... FROM step2;
```

CTEs are evaluated once, named for clarity, and discarded after the query completes. Use them whenever a query needs more than one level of aggregation or filtering.
