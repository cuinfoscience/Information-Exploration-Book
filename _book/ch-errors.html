5  Reading Error Messages

Error messages are not punishments. They are diagnostic information — often the most useful information you will receive when something goes wrong. Learning to read them carefully and systematically is one of the highest-leverage skills you can develop as a programmer. This chapter teaches you how to extract actionable information from Python error messages rather than treating them as walls of intimidating red text.

5.1 Anatomy of a Python Error

When Python encounters a problem, it produces a traceback — a report that shows the sequence of function calls that led to the error, ending with the specific error type and message. Here is a typical example:

---------------------------------------------------------------------------
FileNotFoundError                         Traceback (most recent call last)
Cell In[3], line 1
----> 1 df = pd.read_csv('my_data.csv')

File ~/anaconda3/lib/python3.11/site-packages/pandas/io/parsers/readers.py:948
    948     return _read(filepath_or_buffer, kwds)

FileNotFoundError: [Errno 2] No such file or directory: 'my_data.csv'

Read this from the bottom up. The last line tells you what went wrong: Python could not find a file called my_data.csv. The lines above tell you where the error occurred: in your notebook cell, line 1, in the call to pd.read_csv(). The middle lines show the internal pandas code that was running when the error happened — you can usually skip these.

The two most important pieces of information are always at the bottom: the error type (e.g., FileNotFoundError) and the error message (e.g., No such file or directory: 'my_data.csv').

5.2 Common Error Types

5.2.1 FileNotFoundError

FileNotFoundError: [Errno 2] No such file or directory: 'my_data.csv'

What it means: Python looked for the file at the path you specified and could not find it. This does not mean the file does not exist — it means it does not exist at that location relative to where your code is running.

How to fix it: Check that the file name is spelled correctly, including capitalization. Verify your working directory with import os; print(os.getcwd()). Make sure the file is in the directory you expect. If the file is in a subdirectory, use a relative path like data/raw/my_data.csv.

5.2.2 KeyError

KeyError: 'County'

What it means: You tried to access a column (or dictionary key) that does not exist. This is almost always a spelling or capitalization problem.

How to fix it: Print df.columns.tolist() to see the exact column names, including any hidden whitespace. Common culprits include trailing spaces ('County ' vs 'County'), inconsistent capitalization ('county' vs 'County'), and unexpected characters from the source file.

5.2.3 TypeError

TypeError: unsupported operand type(s) for +: 'int' and 'str'

What it means: You tried to perform an operation on a data type that does not support it — for example, adding a number and a string together. In pandas, this often happens when a column you expected to be numeric is actually stored as text because it contains non-numeric values like $1,234 or N/A.

How to fix it: Check the column’s dtype with df['column'].dtype. If it shows object when you expected a number, inspect the values with df['column'].unique() to find the non-numeric entries, then clean them before converting.

5.2.4 ValueError

ValueError: cannot convert float NaN to integer

What it means: A conversion or operation received a value it cannot handle. This commonly occurs when you try to cast a column containing NaN values to integer type, since regular Python integers cannot represent missing values.

How to fix it: Either fill or drop the NaN values before converting, or use pandas’ nullable integer type (Int64 with a capital I) which can hold missing values.

5.2.5 AttributeError

AttributeError: 'DataFrame' object has no attribute 'str'

What it means: You tried to use a method that does not exist on the object you called it on. The .str accessor, for example, only works on a Series, not on an entire DataFrame.

How to fix it: Check that you are calling the method on the right object. If you meant to apply .str.strip() to a column, make sure you are selecting the column first: df['column'].str.strip(), not df.str.strip().

5.2.6 ModuleNotFoundError

ModuleNotFoundError: No module named 'pandas'

What it means: Python cannot find the package you are trying to import. Either the package is not installed, or your notebook is running in a different environment from the one where you installed it.

How to fix it: Install the package with conda install pandas or pip install pandas. If you already installed it, verify that your notebook is using the correct Python environment (see Chapter 4).

5.3 The DtypeWarning

This is not an error — it is a warning — but it is so common and so important that it deserves special attention:

DtypeWarning: Columns (3) have mixed types. Specify dtype option on import or set low_memory=False.

What it means: Pandas found both numeric and non-numeric values in the same column. It defaulted to storing everything as strings (object type), which means your arithmetic operations will not work as expected.

How to fix it: Investigate the column to find the non-numeric values, then decide whether to clean them or specify the column’s type explicitly at load time with the dtype parameter of read_csv.

5.4 A Systematic Debugging Process

When you encounter an error, resist the urge to randomly change things and re-run. Instead, follow this process:

  1. Read the last line of the traceback. What is the error type? What does the message say?
  2. Read the first arrow line. Which line of your code caused the error?
  3. Reproduce minimally. Can you trigger the same error with a simpler version of the operation? If you were running a complex chain of methods, try each step individually.
  4. Inspect the inputs. Print the shape, dtype, and a sample of the data going into the operation that failed. Is it what you expected?
  5. Search with the error message. Copy the error type and message into a search engine. Someone has almost certainly encountered the same problem.
TipA mindset shift

Beginners often feel that getting an error means they did something wrong. Experienced programmers get errors constantly — the difference is that they read the error message, understand what it is telling them, and use it to fix the problem. Errors are information, not failures.
