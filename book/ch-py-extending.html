8  Extending Data with pandas

Loading and cleaning produce a reliable dataset. Extending transforms that dataset into something that can answer questions. This chapter covers two core operations: creating derived columns from existing data, and computing grouped summaries that collapse many rows into meaningful aggregates. These are the programmatic equivalents of writing formulas and building pivot tables in a spreadsheet — but with the power to handle millions of rows, chain operations together, and reproduce every step exactly.

8.1 Setup

import pandas as pd
import numpy as np

pd.set_option('display.max_columns', None)
pd.set_option('display.float_format', '{:.3f}'.format)

8.2 Datasets

This chapter uses two datasets that illustrate different aspects of extending data.

The Lahman Baseball Database contains historical statistics for every Major League Baseball player and team. Each row in the Batting table represents one player’s performance for one team in one season. We will compute derived metrics like batting average and slugging percentage, then aggregate them across teams and decades.

The Colorado State Demography Office publishes single-year-of-age population estimates by county. Each row represents one county, one year, and one single-year age group. We will bin ages into demographic bands and compute the dependency ratio — a standard population metric.

batting = pd.read_csv('Batting.csv')

print('Shape:', batting.shape)
print('Year range:', batting['yearID'].min(), '–', batting['yearID'].max())
batting.head()
url = 'https://storage.googleapis.com/co-publicdata/sya-county.csv'
pop = pd.read_csv(url, skiprows=1, dtype={'countyfips': str})

print('Shape:', pop.shape)
print('Year range:', pop['year'].min(), '–', pop['year'].max())
print('Datatype values:')
print(pop['datatype'].value_counts())

8.3 Derived Columns

A derived column is a new column computed from existing columns using arithmetic, conditional logic, or binning. The key principle is that derived columns should be transparent — anyone reading your code should be able to see exactly how the new value was computed from the originals.

8.3.1 Arithmetic: Batting Average

Batting average is hits divided by at-bats. The formula is simple, but the implementation requires a decision about what to do when at-bats is zero:

batting['BA'] = batting['H'] / batting['AB'].replace(0, float('nan'))

print('BA stats:')
print(batting['BA'].describe())
print('Rows with NaN BA:', batting['BA'].isna().sum())

Replacing zero with NaN before dividing avoids a ZeroDivisionError and produces NaN for players with no at-bats — which is the correct semantic meaning: their batting average is undefined, not zero.

8.3.2 Multi-step formulas: Slugging Percentage and OBP

More complex metrics require computing intermediate values. Slugging percentage measures batting productivity by weighting different types of hits:

batting['TB'] = (
    batting['H']
    + 2 * batting['2B']
    + 3 * batting['3B']
    + 4 * batting['HR']
)

batting['SLG'] = batting['TB'] / batting['AB'].replace(0, float('nan'))

On-base percentage measures how frequently a batter reaches base. It uses a formula with four variables in the numerator and denominator:

\[OBP = \frac{H + BB + HBP}{AB + BB + HBP + SF}\]

batting['HBP'] = batting['HBP'].fillna(0)
batting['SF'] = batting['SF'].fillna(0)

numerator = batting['H'] + batting['BB'] + batting['HBP']
denominator = batting['AB'] + batting['BB'] + batting['HBP'] + batting['SF']
batting['OBP'] = numerator / denominator

print(batting['OBP'].min(), batting['OBP'].max())

Notice the .fillna(0) calls. The HBP (hit by pitch) and SF (sacrifice fly) columns have missing values for early seasons when those statistics were not tracked. Filling with zero is a documented decision: we are treating “not tracked” as “did not happen.” This is debatable, and the Known Issues Log is where you would note this assumption.

8.3.3 Binning with pd.cut()

Binning transforms a continuous variable into categories. The Colorado population data has single-year ages (0 through 100); we want three demographic bands:

bins = [0, 18, 65, 101]
labels = ['Children (0-17)', 'Working Age (18-64)', 'Seniors (65+)']

pop['age_band'] = pd.cut(pop['age'], bins=bins, labels=labels, right=False)

print('Age band counts:')
print(pop['age_band'].value_counts())

The right=False parameter makes bins left-inclusive: [0, 18) means ages 0 through 17. This puts 18-year-olds in the Working Age band, which is the standard demographic convention.

Always verify that no values fell outside the bins:

print('NaN age bands:', pop['age_band'].isna().sum())

8.3.4 Integer division for decades

A quick way to create decade bins from year values:

batting['decade'] = (batting['yearID'] // 10) * 10

print('Decades:')
print(batting['decade'].value_counts().sort_index())

8.4 Grouped Summaries

The groupby() → agg() pattern is the programmatic equivalent of a spreadsheet pivot table. It splits the data into groups, applies an aggregation function to each group, and combines the results.

8.4.1 Basic groupby: Total HR per Team

latest_year = batting['yearID'].max()
yr_df = batting[batting['yearID'] == latest_year]

team_hr = (
    yr_df
    .groupby('teamID')
    .agg({'HR': 'sum'})
    .reset_index()
    .sort_values('HR', ascending=False)
)

team_hr.head()

8.4.2 Filtering before aggregating

When computing averages, filter to meaningful subsets first. A player with 2 at-bats and 1 hit has a .500 batting average, but that tells you nothing useful:

qualified = batting[batting['AB'] >= 100]

decade_ba = (
    qualified
    .groupby('decade')['BA']
    .agg('mean')
)

decade_ba.plot(title='Mean BA by Decade (players with ≥ 100 AB)')

8.4.3 The dependency ratio: a real-world grouped summary

The dependency ratio — the number of children and seniors per working-age adult — is a standard demographic metric that requires grouped aggregation to compute:

estimates = pop[pop['datatype'] == 'Estimate'].copy()
co_2020 = estimates[estimates['year'] == 2020]

band_summary = co_2020.groupby(['county', 'age_band']).agg(
    pop=('totalpopulation', 'sum')
).reset_index()

# Validate: grouped total should match original
print(f'Grouped total:  {band_summary["pop"].sum():,.0f}')
print(f'Original total: {co_2020["totalpopulation"].sum():,.0f}')

8.5 Three Tools, One Job

Pandas offers three tools for creating cross-tabular summaries, and understanding when to use each saves time and prevents errors.

Tool Aggregates? Use when
groupby().agg() Yes You need flexibility, named aggregations, or long format for further computation
pivot_table() Yes You want aggregation and reshaping in one step (closest to an Excel pivot)
pivot() No Data is already one-row-per-key; you just need to reshape

8.5.1 groupby().agg() — long format output

modern = batting[(batting['yearID'] >= 1990) & (batting['yearID'] <= 2022)].copy()

hr_long = (
    modern
    .groupby(['teamID', 'decade'])['HR']
    .agg('sum')
    .reset_index()
    .rename(columns={'HR': 'total_HR'})
)

print('Shape:', hr_long.shape, '← one row per (team, decade)')
hr_long.head()

8.5.2 pivot_table() — wide format in one step

hr_wide_pt = modern.pivot_table(
    index='teamID',
    columns='decade',
    values='HR',
    aggfunc='sum',
    fill_value=0
)

print('Shape:', hr_wide_pt.shape, '← one row per team, one column per decade')
hr_wide_pt.head()

8.5.3 pivot() — reshape only, no aggregation

pivot() requires every (index, columns) combination to appear exactly once in the data. It will raise a ValueError if there are duplicates, because it cannot aggregate:

# This WORKS because hr_long has one row per (team, decade)
hr_wide_pv = hr_long.pivot(
    index='teamID',
    columns='decade',
    values='total_HR'
)

print('Shape:', hr_wide_pv.shape)

8.5.4 Verifying all three agree

total_long = hr_long['total_HR'].sum()
total_pt = hr_wide_pt.values.sum()
total_pv = hr_wide_pv.fillna(0).values.sum()

print(f'groupby().agg() total HR: {total_long:,}')
print(f'pivot_table()   total HR: {total_pt:,}')
print(f'pivot()         total HR: {total_pv:,}')
print(f'All match: {total_long == total_pt == total_pv}')

8.6 agg() vs. transform()

Both operate on groups, but they produce fundamentally different outputs. The agg() method produces one row per group — the output is smaller than the input. The transform() method broadcasts the group-level result back to every row in the original DataFrame — the output has the same number of rows as the input.

# agg() — summary table
team_totals = (
    batting[batting['yearID'] == 2022]
    .groupby('teamID')['HR']
    .agg('sum')
    .reset_index()
)
print('agg() shape:', team_totals.shape, '(one row per team)')
# transform() — adds group stat to every row
batting['team_total_HR'] = (
    batting
    .groupby(['yearID', 'teamID'])['HR']
    .transform('sum')
)

batting['hr_share'] = batting['HR'] / batting['team_total_HR'].replace(0, float('nan'))

Use transform() when you want to compute something like “what percentage of team home runs did each player contribute?” — a question that requires both the individual value and the group total on the same row.
