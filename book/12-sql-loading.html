12  Loading Data with SQL

This chapter marks the beginning of a new tool — SQL and the SQLite database engine — applied to the same Load–Clean–Extend–Reshape workflow you already know from pandas. Many of the concepts are identical: you are still loading data from files, inspecting it for problems, checking types and counts, and verifying that nothing went wrong. What changes is the interface: instead of calling Python methods on a DataFrame, you write declarative queries that describe what you want, and the database engine figures out how to get it.

SQL (Structured Query Language) has been the standard language for working with databases since the 1970s. You will encounter it in virtually every data-related career. SQLite is a lightweight database engine that stores an entire database in a single file on disk — no server, no installation wizard, no network configuration. This makes it ideal for learning SQL and for working with datasets that fit on a single machine.

The workflow in this chapter moves through four stages: orienting to the terminal, creating a database and schema in the SQLite shell, connecting to the database from Python, and running diagnostic queries with pd.read_sql_query().

12.1 Terminal Orientation

Before touching SQLite, you need three terminal commands. If you completed Chapter 1, these are review.

Command (Mac/Linux) Command (Windows) Purpose
pwd cd (no arguments) Print the current directory
ls dir List files in the current directory
cd folder cd folder Move into a folder

12.1.1 Checking SQLite installation

sqlite3 --version

If you see a version number (e.g., 3.43.2), SQLite is installed and on your PATH. If you get command not found, see Chapter 4 for installation instructions.

12.2 The SQLite Shell

The SQLite shell (sqlite3) is a command-line interface for interacting with a database directly. Think of it as a text-based spreadsheet viewer that also lets you define tables and run queries.

12.2.1 Dot-commands vs. SQL statements

The SQLite shell has two distinct types of commands. Dot-commands start with a period, do not end with a semicolon, and control the shell itself (.tables, .schema, .import, .quit). SQL statements are standard SQL, always end with a semicolon, and operate on data (SELECT ...;, CREATE TABLE ...;).

A SQL statement without a semicolon will sit and wait — SQLite thinks you are still typing. Adding a semicolon after a dot-command will produce an error. Getting these conventions straight early saves considerable frustration.

12.2.2 Step 1: Create the database file

Running sqlite3 cannabis.db either opens an existing file or creates a new one:

sqlite3 cannabis.db ".tables"

The file is created on disk when you first write data to it.

12.2.3 Step 2: Prepare the CSV

ImportantSQL builds on Python

The SQL workflow in this chapter depends on the Python workflow from Chapter 8 and Chapter 9. SQLite’s .import command requires a clean, rectangular CSV — exactly one header row, no aggregate rows, no currency formatting, no merged cells. Real-world government files rarely meet these requirements out of the box. The standard approach is to use pandas to load and clean the raw file first, then export a normalized CSV for SQLite to consume. Students who have not completed Chapter 8 and Chapter 9 should do so before continuing here.

SQLite’s .import command expects a clean, rectangular CSV with exactly one header row and no aggregate rows. The standard approach is to clean the file in pandas first — applying the same operations from Chapter 9 — and export a clean CSV for SQLite to consume.

NotePublic Interest Context: Enclosure by Interface

The fact that you must clean the cannabis file in pandas before SQLite can import it is not a coincidence. Government datasets are typically published through web portals designed for human download — one file at a time, formatted for spreadsheet users. The data is technically open, but the publishing workflow is optimized for a different kind of user than you.

This is a specific form of enclosure: access is possible, but it requires enough technical skill to navigate the formatting. The more agencies publish machine-readable APIs or consistently formatted CSV files with documented schemas, the more equitably public data can be used. See Chapter 7 for the broader framework.

import pandas as pd
import numpy as np

df_raw = pd.read_csv(
    'colorado_cannabis_sales_by_county.csv',
    header=2,        # row index 2 (zero-based) holds the real column headers
    skipfooter=8,    # skip the 8 footnote rows at the bottom of the file
    engine='python', # required when using skipfooter (the C parser doesn't support it)
    na_values=['NR'] # treat "NR" (Not Reported) as a missing value
)

# Rename to clean, SQL-friendly column names (lowercase, no spaces or special chars)
df_raw.columns = ['month', 'year', 'county', 'medical_sales', 'retail_sales']

# Strip currency formatting and cast to nullable integer
for col in ['medical_sales', 'retail_sales']:
    df_raw[col] = (
        df_raw[col]
        .str.strip()                           # remove any surrounding whitespace
        .str.replace('$', '', regex=False)     # remove dollar signs
        .str.replace(',', '', regex=False)     # remove thousands-separator commas
        .pipe(pd.to_numeric, errors='coerce')  # convert strings to numbers; 'coerce' turns failures to NaN
        .astype('Int64')                       # pandas nullable integer type (capital I) — supports NaN
    )                                          # regular int64 does NOT support NaN; Int64 does

# Remove statewide aggregate rows (county = 'Total'); these are not county-level data
df_raw = df_raw[df_raw['county'].str.strip() != 'Total']

# Export to a clean rectangular CSV — one header row, no special formatting
df_raw.to_csv('cannabis_clean.csv', index=False)  # index=False omits the row numbers column
print(f"Exported {len(df_raw):,} rows to cannabis_clean.csv")
TipSQL naming rules

Use lowercase letters, underscores, and no spaces or special characters for column names. The names in the CSV must exactly match the names in the CREATE TABLE schema you will define next. A mismatch causes the header row to be inserted as a data row — one of the most common SQLite loading errors.

12.2.4 Step 3: Define the schema

Before loading data, you declare the table structure — column names and their storage types:

CREATE TABLE IF NOT EXISTS cannabis_sales (
    month INTEGER,
    year INTEGER,
    county TEXT,
    medical_sales INTEGER,
    retail_sales INTEGER
);

SQLite has five storage classes: INTEGER, REAL, TEXT, BLOB, and NULL. For most work, you will use INTEGER, REAL, and TEXT.

12.2.5 Step 4: Load the CSV

The .import command reads a delimited file and inserts rows into a table. Always set .mode csv and .headers on before importing:

sqlite3 cannabis.db << 'EOF'
.mode csv
.headers on
.import cannabis_clean.csv cannabis_sales
EOF
WarningThe heredoc syntax

The << 'EOF' syntax sends multiple lines of input to SQLite through stdin. This is necessary because dot-commands cannot be passed as quoted string arguments on the command line. If you are on Windows, use the Python-side approach described in the next section instead.

12.2.6 Step 5: Post-load diagnostics

Run three checks immediately after loading:

# 1. Row count
sqlite3 cannabis.db "SELECT COUNT(*) AS row_count FROM cannabis_sales;"

# 2. First rows
sqlite3 -column -header cannabis.db "SELECT * FROM cannabis_sales LIMIT 5;"

# 3. Check for accidentally imported header row
sqlite3 cannabis.db "SELECT COUNT(*) AS header_rows FROM cannabis_sales WHERE county = 'county';"

If the header row count is greater than zero, delete it:

DELETE FROM cannabis_sales WHERE county = 'county';

12.3 Connecting from Python

The terminal workflow above is useful for setup and debugging, but your analysis work will happen in Jupyter notebooks using Python’s built-in sqlite3 module and pandas’ pd.read_sql_query().

import sqlite3

conn = sqlite3.connect('cannabis.db')
print("Connection established:", conn)
Warning

If the path is wrong, sqlite3.connect() will silently create a new empty file rather than raising an error. If your queries return “no such table,” check that you are pointing at the right file.

12.3.1 Low-level access with a cursor

A cursor is an object that manages the state of a single database operation — it tracks where you are in the result set, similar to a bookmark in a book. You execute() a query through the cursor, then use fetchone() to retrieve one row at a time, or fetchall() to retrieve all rows as a list of tuples. For most analytical work, pd.read_sql_query() is more convenient because it returns a DataFrame directly.

cursor = conn.cursor()
cursor.execute("SELECT COUNT(*) FROM cannabis_sales;")  # run the query
result = cursor.fetchone()  # retrieve the first (and only) row of results as a tuple
print("Row count:", result[0])  # result is a tuple like (8436,); [0] gets the first element

12.3.2 The preferred workflow: pd.read_sql_query()

This function sends a SQL query and returns a DataFrame — exactly like pd.read_csv(), but reading from a database instead of a file:

df = pd.read_sql_query("SELECT * FROM cannabis_sales LIMIT 10;", conn)
df

The result is a real pandas DataFrame. All the methods you learned in previous chapters — .shape, .dtypes, .describe(), .value_counts() — work exactly the same way.

12.4 SQL Syntax Fundamentals

Every query you write follows a standard clause structure. The clauses must appear in this order:

SELECT   county, year, retail_sales      -- 1. Which columns to return
FROM     cannabis_sales                  -- 2. Which table to read from
WHERE    year = 2023                     -- 3. Which rows to keep (filter)
GROUP BY county                          -- 4. How to collapse rows into groups
HAVING   SUM(retail_sales) > 1000000     -- 5. Filter on group-level results
ORDER BY retail_sales DESC               -- 6. How to sort the output
LIMIT    10;                             -- 7. Maximum rows to return

Not every clause is required. A minimal query needs only SELECT and FROM:

SELECT * FROM cannabis_sales;

12.4.1 Keywords and case

SQL is case-insensitive for keywords — SELECT, select, and Select all work. The convention is to capitalize SQL keywords and use lowercase for table names, column names, and aliases. This helps your eye parse the clause structure quickly.

12.4.2 Literals

When you write a value directly in a query, always use single quotes for text strings ('Boulder'), no quotes for numbers (2023), and NULL for missing values. Double quotes in SQLite are for identifiers (column names), not string values.

12.4.3 The NULL gotcha

NULL = NULL evaluates to NULL, not TRUE. You cannot use = to test for missing values. Always use IS NULL or IS NOT NULL.

12.5 Exploratory Queries

With the data loaded and a Python connection established, apply the post-load checklist using SQL:

# Shape equivalent
pd.read_sql_query("SELECT COUNT(*) AS n_rows FROM cannabis_sales;", conn)
# Head equivalent
pd.read_sql_query("SELECT * FROM cannabis_sales LIMIT 5;", conn)
# Value counts equivalent
pd.read_sql_query("""
    SELECT county, COUNT(*) AS n_months
    FROM cannabis_sales
    GROUP BY county
    ORDER BY n_months DESC
    LIMIT 10;
""", conn)
# Describe equivalent for a numeric column
pd.read_sql_query("""
    SELECT
        MIN(retail_sales) AS min_val,
        MAX(retail_sales) AS max_val,
        AVG(retail_sales) AS mean_val,
        COUNT(*) AS n_rows,
        SUM(CASE WHEN retail_sales IS NULL THEN 1 ELSE 0 END) AS n_nulls
    FROM cannabis_sales;
""", conn)

12.6 SQL ↔︎ pandas Equivalents

pandas SQL
df.shape[0] SELECT COUNT(*) FROM table;
df.head() SELECT * FROM table LIMIT 5;
df['col'].value_counts() SELECT col, COUNT(*) FROM table GROUP BY col ORDER BY COUNT(*) DESC;
df.describe() SELECT MIN(col), MAX(col), AVG(col), COUNT(col) FROM table;
df['col'].isna().sum() SELECT COUNT(*) FROM table WHERE col IS NULL;
df[df['col'] == 'x'] SELECT * FROM table WHERE col = 'x';
df.sort_values('col') SELECT * FROM table ORDER BY col;

| `df[df['col'] == 'x']` | `SELECT * FROM table WHERE col = 'x';` |
| `df.sort_values('col')` | `SELECT * FROM table ORDER BY col;` |
