7  Extending Data with pandas

7.1 Learning Objectives

By the end of this chapter, you will be able to:

  • Create derived columns using arithmetic, conditional logic, and pd.cut binning.
  • Parse dates from integer columns using pd.to_datetime and apply time-series operations like resampling.
  • Compute grouped summaries using df.groupby() with .agg().
  • Choose between groupby().agg(), pivot_table, and pivot based on whether aggregation and reshaping are needed.
  • Distinguish .agg() (one row per group) from .transform() (same shape as input) and pick the right tool for a given question.

7.2 Why Extending Matters

Loading and cleaning produce a reliable dataset. Extending transforms that dataset into something that can answer questions. This chapter covers two core operations: creating derived columns from existing data, and computing grouped summaries that collapse many rows into meaningful aggregates. These are the programmatic equivalents of writing formulas and building pivot tables in a spreadsheet — but with the power to handle millions of rows, chain operations together, and reproduce every step exactly.

This chapter assumes you have completed Chapter 6. The batting and population data used here must be loaded and cleaned before extending.

7.5 Derived Columns

A derived column is a new column computed from existing columns using arithmetic, conditional logic, or binning. The key principle is that derived columns should be transparent — anyone reading your code should be able to see exactly how the new value was computed from the originals.

7.5.1 Arithmetic: Batting Average

Batting average is hits divided by at-bats. The formula is simple, but the implementation requires a decision about what to do when at-bats is zero:

batting['BA'] = batting['H'] / batting['AB'].replace(0, float('nan'))

print('BA stats:')
print(batting['BA'].describe())
print('Rows with NaN BA:', batting['BA'].isna().sum())

Replacing zero with NaN before dividing avoids a ZeroDivisionError and produces NaN for players with no at-bats — which is the correct semantic meaning: their batting average is undefined, not zero.

7.5.2 Multi-step formulas: Slugging Percentage and OBP

More complex metrics require computing intermediate values. Slugging percentage measures batting productivity by weighting different types of hits:

batting['TB'] = (
    batting['H']
    + 2 * batting['2B']
    + 3 * batting['3B']
    + 4 * batting['HR']
)

batting['SLG'] = batting['TB'] / batting['AB'].replace(0, float('nan'))

On-base percentage measures how frequently a batter reaches base. It uses a formula with four variables in the numerator and denominator:

\[OBP = \frac{H + BB + HBP}{AB + BB + HBP + SF}\]

batting['HBP'] = batting['HBP'].fillna(0)
batting['SF'] = batting['SF'].fillna(0)

numerator = batting['H'] + batting['BB'] + batting['HBP']
denominator = batting['AB'] + batting['BB'] + batting['HBP'] + batting['SF']
batting['OBP'] = numerator / denominator

print(batting['OBP'].min(), batting['OBP'].max())

Notice the .fillna(0) calls. The HBP (hit by pitch) and SF (sacrifice fly) columns have missing values for early seasons when those statistics were not tracked. Filling with zero is a documented decision: we are treating “not tracked” as “did not happen.” This is debatable, and the Known Issues Log is where you would note this assumption.

7.5.3 Binning with pd.cut()

Binning transforms a continuous variable into categories. The Colorado population data has single-year ages (0 through 100); we want three demographic bands:

bins = [0, 18, 65, 101]
labels = ['Children (0-17)', 'Working Age (18-64)', 'Seniors (65+)']

pop['age_band'] = pd.cut(pop['age'], bins=bins, labels=labels, right=False)

print('Age band counts:')
print(pop['age_band'].value_counts())

The right=False parameter makes bins left-inclusive: [0, 18) means ages 0 through 17. This puts 18-year-olds in the Working Age band, which is the standard demographic convention.

Always verify that no values fell outside the bins:

print('NaN age bands:', pop['age_band'].isna().sum())

7.5.4 Integer division for decades

A quick way to create decade bins from year values:

batting['decade'] = (batting['yearID'] // 10) * 10

print('Decades:')
print(batting['decade'].value_counts().sort_index())

7.5.5 Parsing Dates: Cannabis Sales

Government datasets frequently store dates as separate integer columns — Month and Year rather than a single date column. The cannabis sales data is a typical example. pd.to_datetime() can reconstruct a proper datetime from integer month and year values:

cannabis = pd.read_csv(
    'colorado_cannabis_sales_by_county.csv',
    header=2,
    skipfooter=8,
    engine='python',
    na_values=['NR']
)
cannabis.columns = ['Month', 'Year', 'County', 'Medical', 'Retail']
cannabis['County'] = cannabis['County'].str.strip()
for col in ['Medical', 'Retail']:
    cannabis[col] = pd.to_numeric(
        cannabis[col].str.replace('$', '', regex=False).str.replace(',', '', regex=False)
    )
cannabis = cannabis[cannabis['County'] != 'Total']

# Construct a date column from integer Month and Year
cannabis['date'] = pd.to_datetime(
    cannabis[['Year', 'Month']].assign(Day=1).rename(
        columns={'Year': 'year', 'Month': 'month'}
    )
)

print(cannabis[['Month', 'Year', 'date']].head())

pd.to_datetime() accepts a DataFrame with year, month, and day columns. The .assign(Day=1) adds a constant day-of-month, and .rename() converts the column names to lowercase as required.

Once parsed as a proper datetime, the column unlocks time-series operations — sorting chronologically, computing time differences, and resampling by period:

# Quarterly retail sales, statewide
quarterly = (
    cannabis
    .set_index('date')
    ['Retail']
    .resample('QE')
    .sum()
)

print(quarterly.tail(8))
TipStoring dates as integers is not a mistake

Storing month and year as separate integer columns is a deliberate design in many government systems — it makes SQL filtering simple (WHERE year = 2022) without requiring date functions. When you encounter this pattern, pd.to_datetime() with assign(Day=1) is the standard reconstruction approach.

7.6 Grouped Summaries

The groupby() → agg() pattern is the programmatic equivalent of a spreadsheet pivot table. It splits the data into groups, applies an aggregation function to each group, and combines the results.

7.6.1 Basic groupby: Total HR per Team

latest_year = batting['yearID'].max()
yr_df = batting[batting['yearID'] == latest_year]

team_hr = (
    yr_df
    .groupby('teamID')
    .agg({'HR': 'sum'})
    .reset_index()
    .sort_values('HR', ascending=False)
)

team_hr.head()

7.6.2 Filtering before aggregating

When computing averages, filter to meaningful subsets first. A player with 2 at-bats and 1 hit has a .500 batting average, but that tells you nothing useful:

qualified = batting[batting['AB'] >= 100]

decade_ba = (
    qualified
    .groupby('decade')['BA']
    .agg('mean')
)

decade_ba.plot(title='Mean BA by Decade (players with ≥ 100 AB)')

7.6.3 The dependency ratio: a real-world grouped summary

The dependency ratio — the number of children and seniors per working-age adult — is a standard demographic metric that requires grouped aggregation to compute:

estimates = pop[pop['datatype'] == 'Estimate'].copy()
co_2020 = estimates[estimates['year'] == 2020]

band_summary = co_2020.groupby(['county', 'age_band']).agg(
    pop=('totalpopulation', 'sum')
).reset_index()

# Validate: grouped total should match original
print(f'Grouped total:  {band_summary["pop"].sum():,.0f}')
print(f'Original total: {co_2020["totalpopulation"].sum():,.0f}')

7.7 Three Tools, One Job

Pandas offers three tools for creating cross-tabular summaries, and understanding when to use each saves time and prevents errors.

Tool Aggregates? Use when
groupby().agg() Yes You need flexibility, named aggregations, or long format for further computation
pivot_table() Yes You want aggregation and reshaping in one step (closest to an Excel pivot)
pivot() No Data is already one-row-per-key; you only need to reshape

7.7.1 groupby().agg() — long format output

modern = batting[(batting['yearID'] >= 1990) & (batting['yearID'] <= 2022)].copy()

hr_long = (
    modern
    .groupby(['teamID', 'decade'])['HR']
    .agg('sum')
    .reset_index()
    .rename(columns={'HR': 'total_HR'})
)

print('Shape:', hr_long.shape, '← one row per (team, decade)')
hr_long.head()

7.7.2 pivot_table() — wide format in one step

hr_wide_pt = modern.pivot_table(
    index='teamID',
    columns='decade',
    values='HR',
    aggfunc='sum',
    fill_value=0
)

print('Shape:', hr_wide_pt.shape, '← one row per team, one column per decade')
hr_wide_pt.head()

7.7.3 pivot() — reshape only, no aggregation

pivot() requires every (index, columns) combination to appear exactly once in the data. It will raise a ValueError if there are duplicates, because it cannot aggregate:

# This WORKS because hr_long has one row per (team, decade)
hr_wide_pv = hr_long.pivot(
    index='teamID',
    columns='decade',
    values='total_HR'
)

print('Shape:', hr_wide_pv.shape)

7.7.4 Verifying all three agree

total_long = hr_long['total_HR'].sum()
total_pt = hr_wide_pt.values.sum()
total_pv = hr_wide_pv.fillna(0).values.sum()

print(f'groupby().agg() total HR: {total_long:,}')
print(f'pivot_table()   total HR: {total_pt:,}')
print(f'pivot()         total HR: {total_pv:,}')
print(f'All match: {total_long == total_pt == total_pv}')

7.8 agg() vs. transform()

Both operate on groups, but they produce fundamentally different outputs. The agg() method produces one row per group — the output is smaller than the input. The transform() method broadcasts the group-level result back to every row in the original DataFrame — the output has the same number of rows as the input.

# agg() — summary table
team_totals = (
    batting[batting['yearID'] == 2022]
    .groupby('teamID')['HR']
    .agg('sum')
    .reset_index()
)
print('agg() shape:', team_totals.shape, '(one row per team)')
# transform() — adds group stat to every row
batting['team_total_HR'] = (
    batting
    .groupby(['yearID', 'teamID'])['HR']
    .transform('sum')
)

batting['hr_share'] = batting['HR'] / batting['team_total_HR'].replace(0, float('nan'))

Use transform() when you want to compute something like “what percentage of team home runs did each player contribute?” — a question that requires both the individual value and the group total on the same row.

7.9 Public Interest Context: Oversight

Derived metrics are interpretive choices dressed up as arithmetic. A rate is a ratio — and the denominator you pick determines the story. “Deaths per 100,000 residents” and “deaths per 100,000 adults” and “deaths per 100,000 people with health insurance” can all point the same way or the opposite way on the same underlying data. A dependency ratio that excludes seniors under 65 who are still working looks different from one that doesn’t. When you create a derived column, you are quietly asserting that this is the right way to compare groups, and your assertion travels downstream into every chart and conclusion.

Grouped summaries have the same property. A groupby that silently drops rows with NaN in the group column, or a bin that accidentally truncates values outside its edges, is hidden data loss that still produces a confident-looking table. Oversight in extending data means auditing the arithmetic: re-deriving totals, comparing group sums to source sums, and explicitly naming the denominator you chose and why. When you hand someone a rate table, can they reconstruct the formula you used to build it?

NoteConnection to Chapter 4

This section connects to the concept of oversight introduced in Chapter 4. Every derived metric is an analytical decision; oversight requires that those decisions be explicit and auditable rather than buried in a one-line arithmetic expression.

7.10 Common Issues in This Chapter

WarningZeroDivisionError when computing a rate

What it means: You divided by a column that contains zero. Batting average = hits / at-bats fails when at-bats is 0.

How to fix it: Replace the divisor’s zeros with NaN before dividing: df['H'] / df['AB'].replace(0, float('nan')). This produces NaN for undefined cases, which is the correct semantic.

Warningpivot() raises ValueError: Index contains duplicate entries

What it means: pivot() does not aggregate — it requires each (index, columns) combination to appear exactly once. If your source has multiple rows per combination, pivot() cannot decide which value to use.

How to fix it: Use pivot_table() instead, which takes an aggfunc parameter. Or aggregate first with groupby().agg() so each combination is unique, then call pivot().

Warninggroupby aggregation produces unexpected NaN

What it means: Groups with all-missing input produce NaN in the output, which is easy to misread as a zero.

How to fix it: Decide what missing should mean for your question. For counts, fill_value=0 in pivot_table or .fillna(0) after aggregation is usually appropriate. For means, leaving NaN is often correct — averaging nothing is not zero.

WarningGrouped totals do not match the original total

What it means: Either rows were filtered before grouping, or a groupby column has NaN values (which are excluded by default).

How to fix it: Compare df['col'].sum() before grouping and grouped['col'].sum() after. If they differ, inspect df['group_col'].isna().sum(). Use dropna=False in groupby if you want to keep missing-group rows in the output.

7.11 Key Takeaways

  • Derived columns should be transparent: reading the code should tell you exactly how a new value was computed. Avoid magic numbers and undocumented assumptions.
  • groupby().agg() → long format; pivot_table() → wide format in one step; pivot() → reshape only (no aggregation). Pick the tool that matches the question.
  • .agg() collapses a group to one row. .transform() broadcasts the group result back to every row. Use .transform() when the individual value and the group stat need to coexist.
  • Validate every aggregation by comparing totals before and after. Silent row drops — from NaN in a groupby key, or from missing data — are the most common source of wrong answers.
  • Binning continuous variables with pd.cut is a standard way to turn ages, incomes, or scores into analytically meaningful categories.

### Parsing Dates: Cannabis Sales

Government datasets frequently store dates as separate integer columns — `Month` and `Year` rather than a single date column. The cannabis sales data is a typical example. `pd.to_datetime()` can reconstruct a proper datetime from integer month and year values:

```{python}
cannabis = pd.read_csv(
    'colorado_cannabis_sales_by_county.csv',
    header=2,
    skipfooter=8,
    engine='python',
    na_values=['NR']
)
cannabis.columns = ['Month', 'Year', 'County', 'Medical', 'Retail']
cannabis['County'] = cannabis['County'].str.strip()
for col in ['Medical', 'Retail']:
    cannabis[col] = pd.to_numeric(
        cannabis[col].str.replace('$', '', regex=False).str.replace(',', '', regex=False)
    )
cannabis = cannabis[cannabis['County'] != 'Total']

# Construct a date column from integer Month and Year
cannabis['date'] = pd.to_datetime(
    cannabis[['Year', 'Month']].assign(Day=1).rename(
        columns={'Year': 'year', 'Month': 'month'}
    )
)

print(cannabis[['Month', 'Year', 'date']].head())
```

`pd.to_datetime()` accepts a DataFrame with `year`, `month`, and `day` columns. The `.assign(Day=1)` adds a constant day-of-month, and `.rename()` converts the column names to lowercase as required.

Once parsed as a proper datetime, the column unlocks time-series operations — sorting chronologically, computing time differences, and resampling by period:

```{python}
# Quarterly retail sales, statewide
quarterly = (
    cannabis
    .set_index('date')
    ['Retail']
    .resample('QE')
    .sum()
)

print(quarterly.tail(8))
```

::: {.callout-tip}
## Storing dates as integers is not a mistake

Storing month and year as separate integer columns is a deliberate design in many government systems — it makes SQL filtering simple (`WHERE year = 2022`) without requiring date functions. When you encounter this pattern, `pd.to_datetime()` with `assign(Day=1)` is the standard reconstruction approach.
:::

## Grouped Summaries

The `groupby()` → `agg()` pattern is the programmatic equivalent of a spreadsheet pivot table. It splits the data into groups, applies an aggregation function to each group, and combines the results.

### Basic groupby: Total HR per Team

```{python}
latest_year = batting['yearID'].max()
yr_df = batting[batting['yearID'] == latest_year]

team_hr = (
    yr_df
    .groupby('teamID')
    .agg({'HR': 'sum'})
    .reset_index()
    .sort_values('HR', ascending=False)
)

team_hr.head()
```

### Filtering before aggregating

When computing averages, filter to meaningful subsets first. A player with 2 at-bats and 1 hit has a .500 batting average, but that tells you nothing useful:

```{python}
qualified = batting[batting['AB'] >= 100]

decade_ba = (
    qualified
    .groupby('decade')['BA']
    .agg('mean')
)

decade_ba.plot(title='Mean BA by Decade (players with ≥ 100 AB)')
```

### The dependency ratio: a real-world grouped summary

The dependency ratio — the number of children and seniors per working-age adult — is a standard demographic metric that requires grouped aggregation to compute:

```{python}
estimates = pop[pop['datatype'] == 'Estimate'].copy()
co_2020 = estimates[estimates['year'] == 2020]

band_summary = co_2020.groupby(['county', 'age_band']).agg(
    pop=('totalpopulation', 'sum')
).reset_index()

# Validate: grouped total should match original
print(f'Grouped total:  {band_summary["pop"].sum():,.0f}')
print(f'Original total: {co_2020["totalpopulation"].sum():,.0f}')
```

## Three Tools, One Job

Pandas offers three tools for creating cross-tabular summaries, and understanding when to use each saves time and prevents errors.

| Tool | Aggregates? | Use when |
|------|------------|----------|
| `groupby().agg()` | Yes | You need flexibility, named aggregations, or long format for further computation |
| `pivot_table()` | Yes | You want aggregation and reshaping in one step (closest to an Excel pivot) |
| `pivot()` | **No** | Data is already one-row-per-key; you only need to reshape |

### `groupby().agg()` — long format output

```{python}
modern = batting[(batting['yearID'] >= 1990) & (batting['yearID'] <= 2022)].copy()

hr_long = (
    modern
    .groupby(['teamID', 'decade'])['HR']
    .agg('sum')
    .reset_index()
    .rename(columns={'HR': 'total_HR'})
)

print('Shape:', hr_long.shape, '← one row per (team, decade)')
hr_long.head()
```

### `pivot_table()` — wide format in one step

```{python}
hr_wide_pt = modern.pivot_table(
    index='teamID',
    columns='decade',
    values='HR',
    aggfunc='sum',
    fill_value=0
)

print('Shape:', hr_wide_pt.shape, '← one row per team, one column per decade')
hr_wide_pt.head()
```

### `pivot()` — reshape only, no aggregation

`pivot()` requires every `(index, columns)` combination to appear exactly once in the data. It will raise a `ValueError` if there are duplicates, because it cannot aggregate:

```{python}
# This WORKS because hr_long has one row per (team, decade)
hr_wide_pv = hr_long.pivot(
    index='teamID',
    columns='decade',
    values='total_HR'
)

print('Shape:', hr_wide_pv.shape)
```

### Verifying all three agree

```{python}
total_long = hr_long['total_HR'].sum()
total_pt = hr_wide_pt.values.sum()
total_pv = hr_wide_pv.fillna(0).values.sum()

print(f'groupby().agg() total HR: {total_long:,}')
print(f'pivot_table()   total HR: {total_pt:,}')
print(f'pivot()         total HR: {total_pv:,}')
print(f'All match: {total_long == total_pt == total_pv}')
```

## `agg()` vs. `transform()`

Both operate on groups, but they produce fundamentally different outputs. The `agg()` method produces one row per group — the output is smaller than the input. The `transform()` method broadcasts the group-level result back to every row in the original DataFrame — the output has the same number of rows as the input.

```{python}
# agg() — summary table
team_totals = (
    batting[batting['yearID'] == 2022]
    .groupby('teamID')['HR']
    .agg('sum')
    .reset_index()
)
print('agg() shape:', team_totals.shape, '(one row per team)')
```

```{python}
# transform() — adds group stat to every row
batting['team_total_HR'] = (
    batting
    .groupby(['yearID', 'teamID'])['HR']
    .transform('sum')
)

batting['hr_share'] = batting['HR'] / batting['team_total_HR'].replace(0, float('nan'))
```

Use `transform()` when you want to compute something like "what percentage of team home runs did each player contribute?" — a question that requires both the individual value and the group total on the same row.

## Public Interest Context: Oversight

Derived metrics are interpretive choices dressed up as arithmetic. A rate is a ratio — and the denominator you pick determines the story. "Deaths per 100,000 residents" and "deaths per 100,000 adults" and "deaths per 100,000 people with health insurance" can all point the same way or the opposite way on the same underlying data. A dependency ratio that excludes seniors under 65 who are still working looks different from one that doesn't. When you create a derived column, you are quietly asserting that this is the right way to compare groups, and your assertion travels downstream into every chart and conclusion.

Grouped summaries have the same property. A `groupby` that silently drops rows with NaN in the group column, or a bin that accidentally truncates values outside its edges, is hidden data loss that still produces a confident-looking table. Oversight in extending data means auditing the arithmetic: re-deriving totals, comparing group sums to source sums, and explicitly naming the denominator you chose and why. When you hand someone a rate table, can they reconstruct the formula you used to build it?

This section connects to the concept of **oversight** introduced in [Chapter 4](04-public-interest.qmd). Every derived metric is an analytical decision; oversight requires that those decisions be explicit and auditable rather than buried in a one-line arithmetic expression.

## Common Issues in This Chapter

::: {.callout-warning}
## `ZeroDivisionError` when computing a rate
**What it means:** You divided by a column that contains zero. Batting average = hits / at-bats fails when at-bats is 0.

**How to fix it:** Replace the divisor's zeros with `NaN` before dividing: `df['H'] / df['AB'].replace(0, float('nan'))`. This produces `NaN` for undefined cases, which is the correct semantic.
:::

::: {.callout-warning}
## `pivot()` raises `ValueError: Index contains duplicate entries`
**What it means:** `pivot()` does not aggregate — it requires each `(index, columns)` combination to appear exactly once. If your source has multiple rows per combination, `pivot()` cannot decide which value to use.

**How to fix it:** Use `pivot_table()` instead, which takes an `aggfunc` parameter. Or aggregate first with `groupby().agg()` so each combination is unique, then call `pivot()`.
:::

::: {.callout-warning}
## `groupby` aggregation produces unexpected NaN
**What it means:** Groups with all-missing input produce `NaN` in the output, which is easy to misread as a zero.

**How to fix it:** Decide what missing should mean for your question. For counts, `fill_value=0` in `pivot_table` or `.fillna(0)` after aggregation is usually appropriate. For means, leaving `NaN` is often correct — averaging nothing is not zero.
:::

::: {.callout-warning}
## Grouped totals do not match the original total
**What it means:** Either rows were filtered before grouping, or a `groupby` column has `NaN` values (which are excluded by default).

**How to fix it:** Compare `df['col'].sum()` before grouping and `grouped['col'].sum()` after. If they differ, inspect `df['group_col'].isna().sum()`. Use `dropna=False` in `groupby` if you want to keep missing-group rows in the output.
:::

## Key Takeaways

- Derived columns should be **transparent**: reading the code should tell you exactly how a new value was computed. Avoid magic numbers and undocumented assumptions.
- `groupby().agg()` → long format; `pivot_table()` → wide format in one step; `pivot()` → reshape only (no aggregation). Pick the tool that matches the question.
- `.agg()` collapses a group to one row. `.transform()` broadcasts the group result back to every row. Use `.transform()` when the individual value and the group stat need to coexist.
- Validate every aggregation by comparing totals before and after. Silent row drops — from `NaN` in a groupby key, or from missing data — are the most common source of wrong answers.
- Binning continuous variables with `pd.cut` is a standard way to turn ages, incomes, or scores into analytically meaningful categories.

## Exercise 3: Compare `pivot_table()` and `groupby()` → `pivot()` paths
Using the dependency ratio data from Exercise 2, implement the same result using the two-step `groupby().agg()` → `pivot()` path. Verify that both approaches produce identical results using `pd.testing.assert_frame_equal()`.
