3  Documentation and Markdown

3.1 Learning Objectives

By the end of this chapter, you will be able to:

  • Write basic Markdown — headings, emphasis, lists, links, code blocks, and tables — in notebooks, README files, and GitHub.
  • Structure a notebook with alternating narrative Markdown and code cells using the goal/method/interpretation pattern.
  • Maintain a Known Issues Log that records each cleaning decision with problem, fix, impact, and remaining uncertainty.
  • Build a Data Dictionary that describes every column, its type, its valid values, and the cleaning applied.
  • Write code comments that explain why rather than what.

3.2 Why Documentation Matters

The most technically brilliant analysis is worthless if no one — including your future self — can understand what it does or why it does it. Documentation is not an afterthought you add when the work is finished; it is a running record you maintain while the work is happening. This chapter covers the tools and habits that make documentation practical rather than burdensome.

3.3 Why You Document As You Go

There is a persistent fantasy in data work that you will finish the analysis first and document it later. In practice, “later” never comes, and even when it does, you have forgotten the reasoning behind your decisions. Why did you drop those 23 rows? Why did you recode “OTH” as missing instead of keeping it? Why did you use a left join instead of an inner join?

These decisions are judgment calls, not mechanical steps. If you do not record them at the time you make them, they are lost. The solution is to build documentation into your workflow so that it requires almost no extra effort.

3.4 Markdown Basics

Markdown is a lightweight formatting language used in Jupyter notebooks, README files, GitHub, and many other tools. You write plain text with a few simple markers, and the rendering engine converts it into formatted output.

3.4.1 Headings

Use # symbols to create headings. More # symbols mean smaller headings:

# Top-Level Heading
## Section Heading
### Subsection Heading

3.4.2 Emphasis

Wrap text in asterisks for emphasis:

*italic text*
**bold text**
`inline code`

3.4.3 Lists

Create unordered lists with dashes or asterisks, and ordered lists with numbers:

- First item
- Second item
- Third item

1. Step one
2. Step two
3. Step three

3.4.4 Links

[Link text](https://example.com)

3.4.5 Code blocks

For multi-line code, use triple backticks with an optional language identifier:

```python
df = pd.read_csv('data.csv')
df.head()
```

3.4.6 Tables

| Column | Type | Description |
|--------|------|-------------|
| county | str  | County name |
| year   | int  | Calendar year |

These few elements cover the vast majority of what you need for data documentation. You do not need to learn the full Markdown specification — these basics are enough.

3.5 Documentation in Jupyter Notebooks

A Jupyter notebook interleaves Markdown cells with code cells. This makes it the ideal environment for literate analysis, where your explanations sit directly alongside the code they describe.

3.5.1 Narrative Markdown cells

Before each code section, add a Markdown cell that explains what you are about to do and why. After a code section that produces output, add a cell that interprets the result. This narrative structure turns a notebook from a collection of code snippets into a readable analysis.

A useful pattern is to frame each section with three elements: the goal (what question are you answering?), the method (what operation are you performing?), and the interpretation (what did you learn from the result?).

3.5.2 The Known Issues Log

One of the most valuable documentation habits is maintaining a running log of problems you discover and decisions you make during cleaning. The Known Issues Log is a Markdown cell — typically placed near the top or bottom of your notebook — that records each issue with a consistent structure:

## Known Issues and Cleaning Decisions

### 2026-02-17: Party field standardization
**Problem:** Party stored inconsistently — "Dem", "DEM", "D", "Democrat"
all mean the same thing.
**Fix:** Created party_map dictionary, normalized to "DEM", "REP", "UNA".
**Impact:** Reduced unique values from 47 to 3; no rows lost.
**Remaining issue:** 23 rows have party="OTH" — unclear what this means.

Each entry records four things: the problem, the fix, the impact (especially any data loss), and any remaining uncertainty. This log becomes essential when you revisit the analysis weeks later or when someone else needs to understand your decisions.

NotePublic Interest Context: Openness

The Known Issues Log is a transparency tool. In public interest data work, publishing your data problems alongside your conclusions is what openness looks like in practice. It is the difference between presenting findings as if they emerged cleanly from perfect data and acknowledging that every real-world dataset has quirks, and that you made judgment calls about how to handle them.

Openness does not mean you must apologize for your data. Every analysis has limitations. It means you make your limitations visible so that readers can evaluate them — and so that you can be held accountable for the decisions you made. A Known Issues Log entry that says “23 rows with party='OTH' were excluded because the category is ambiguous” is far more honest and defensible than leaving the exclusion unmentioned.

See Chapter 7 for the broader framework connecting openness to public interest data practice.

3.5.3 The Data Dictionary

A data dictionary describes what each column in your dataset contains. It is the single most useful piece of documentation you can write, and it takes only a few minutes. Here is a Markdown table format that works well in notebooks:

| Column | Type | Description | Valid Values | Cleaning Applied |
|--------|------|-------------|--------------|------------------|
| county | str  | County name | 64 CO counties | .str.strip().str.upper() |
| year   | int  | Calendar year | 2014–2024 | pd.to_numeric() |
| sales  | float | Sales in USD | >= 0 | Removed $ and commas |

The data dictionary should be updated whenever you add derived columns or recode values. It is a living document, not a one-time artifact.

3.6 Comments in Code

Markdown cells handle the narrative layer of documentation. Code comments handle the technical layer — explaining why a particular line of code does what it does, not what it does (which should be clear from the code itself).

# Good comment: explains WHY
# Replace 0 AB with NaN to avoid division-by-zero in batting average
batting['BA'] = batting['H'] / batting['AB'].replace(0, float('nan'))

# Bad comment: restates WHAT the code does
# Divide H by AB
batting['BA'] = batting['H'] / batting['AB']

A useful heuristic: if the code does exactly what it says — df.head(), print(x) — it does not need a comment. If the code involves a non-obvious decision — why you chose errors='coerce' instead of errors='raise', why you filtered to rows with more than 100 at-bats, why you used a left join instead of an inner join — that is worth a comment.

3.7 Common Issues in This Chapter

WarningYour Markdown renders as raw text instead of formatted output

What it means: The cell in Jupyter is set to “Code” instead of “Markdown,” so the asterisks and pound signs appear literally.

How to fix it: Select the cell and press M (when in command mode, not edit mode) or use the dropdown at the top of the notebook to switch it to Markdown. Re-run the cell with Shift+Enter.

WarningYour Known Issues Log gets out of sync with the code

What it means: You cleaned something, then forgot to log it — weeks later you can’t tell why your row count changed.

How to fix it: Build the habit of writing the log entry before running the cleaning cell, not after. The log entry is a plan; the code is the execution.

WarningYour notebook is all code with no narrative

What it means: You have a working analysis but no one reading it — including you in three months — can tell what question you were answering or what you concluded.

How to fix it: Add a top-level Markdown cell with the research question. Before each section of code, add a one-sentence Markdown cell that states the goal. After each result, add a one-sentence interpretation. You will be surprised how much this helps.

3.8 Key Takeaways

  • Documentation is a habit, not a final step. The Known Issues Log and Data Dictionary are living documents you maintain while working.
  • Markdown is tiny — headings, emphasis, lists, links, code blocks, tables — and covers almost everything you need for data documentation.
  • Code comments should explain why you made a decision, not what a line of code does.
  • A notebook with interleaved narrative and code tells a story. A notebook with only code is a puzzle your future self has to solve.

3.9 Further Reading

Tip📚 Going deeper on documentation and Markdown
  • Markdown Guide — markdownguide.org is the cleanest reference for the syntax. The “Cheat Sheet” page is worth bookmarking.
  • GitHub Flavored Markdown Spec — github.github.com/gfm/ documents GitHub’s specific Markdown dialect, including task lists and tables.
  • Jupyter Notebook Markdown cells — jupyter-notebook.readthedocs.io describes how Markdown is rendered inside notebooks.
  • “The Practice of Reproducible Research” — a free collection of case studies at practicereproducibleresearch.org shows how working data scientists document their decisions.
