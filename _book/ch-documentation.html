3  Documentation and Markdown

The most technically brilliant analysis is worthless if no one — including your future self — can understand what it does or why it does it. Documentation is not an afterthought you add when the work is finished; it is a running record you maintain while the work is happening. This chapter covers the tools and habits that make documentation practical rather than burdensome.

3.1 Why You Document As You Go

There is a persistent fantasy in data work that you will finish the analysis first and document it later. In practice, “later” never comes, and even when it does, you have forgotten the reasoning behind your decisions. Why did you drop those 23 rows? Why did you recode “OTH” as missing instead of keeping it? Why did you use a left join instead of an inner join?

These decisions are judgment calls, not mechanical steps. If you do not record them at the time you make them, they are lost. The solution is to build documentation into your workflow so that it requires almost no extra effort.

3.2 Markdown Basics

Markdown is a lightweight formatting language used in Jupyter notebooks, README files, GitHub, and many other tools. You write plain text with a few simple markers, and the rendering engine converts it into formatted output.

3.2.1 Headings

Use # symbols to create headings. More # symbols mean smaller headings:

# Top-Level Heading
## Section Heading
### Subsection Heading

3.2.2 Emphasis

Wrap text in asterisks for emphasis:

*italic text*
**bold text**
`inline code`

3.2.3 Lists

Create unordered lists with dashes or asterisks, and ordered lists with numbers:

3.2.5 Code blocks

For multi-line code, use triple backticks with an optional language identifier:

```python
df = pd.read_csv('data.csv')
df.head()
```

3.2.6 Tables

| Column | Type | Description |
|--------|------|-------------|
| county | str  | County name |
| year   | int  | Calendar year |

These few elements cover the vast majority of what you need for data documentation. You do not need to learn the full Markdown specification — just these basics.

3.3 Documentation in Jupyter Notebooks

A Jupyter notebook interleaves Markdown cells with code cells. This makes it the ideal environment for literate analysis, where your explanations sit directly alongside the code they describe.

3.3.1 Narrative Markdown cells

Before each code section, add a Markdown cell that explains what you are about to do and why. After a code section that produces output, add a cell that interprets the result. This narrative structure turns a notebook from a collection of code snippets into a readable analysis.

A useful pattern is to frame each section with three elements: the goal (what question are you answering?), the method (what operation are you performing?), and the interpretation (what did you learn from the result?).

3.3.2 The Known Issues Log

One of the most valuable documentation habits is maintaining a running log of problems you discover and decisions you make during cleaning. The Known Issues Log is a Markdown cell — typically placed near the top or bottom of your notebook — that records each issue with a consistent structure:

## Known Issues and Cleaning Decisions

### 2026-02-17: Party field standardization
**Problem:** Party stored inconsistently — "Dem", "DEM", "D", "Democrat"
all mean the same thing.
**Fix:** Created party_map dictionary, normalized to "DEM", "REP", "UNA".
**Impact:** Reduced unique values from 47 to 3; no rows lost.
**Remaining issue:** 23 rows have party="OTH" — unclear what this means.

Each entry records four things: the problem, the fix, the impact (especially any data loss), and any remaining uncertainty. This log becomes essential when you revisit the analysis weeks later or when someone else needs to understand your decisions.

3.3.3 The Data Dictionary

A data dictionary describes what each column in your dataset contains. It is the single most useful piece of documentation you can write, and it takes only a few minutes. Here is a Markdown table format that works well in notebooks:

| Column | Type | Description | Valid Values | Cleaning Applied |
|--------|------|-------------|--------------|------------------|
| county | str  | County name | 64 CO counties | .str.strip().str.upper() |
| year   | int  | Calendar year | 2014–2024 | pd.to_numeric() |
| sales  | float | Sales in USD | >= 0 | Removed $ and commas |

The data dictionary should be updated whenever you add derived columns or recode values. It is a living document, not a one-time artifact.

3.4 Comments in Code

Markdown cells handle the narrative layer of documentation. Code comments handle the technical layer — explaining why a particular line of code does what it does, not what it does (which should be clear from the code itself).

# Good comment: explains WHY
# Replace 0 AB with NaN to avoid division-by-zero in batting average
batting['BA'] = batting['H'] / batting['AB'].replace(0, float('nan'))

# Bad comment: restates WHAT the code does
# Divide H by AB
batting['BA'] = batting['H'] / batting['AB']

A useful heuristic: if the code is straightforward, you do not need a comment. If the code involves a non-obvious decision — why you chose errors='coerce' instead of errors='raise', why you filtered to rows with more than 100 at-bats, why you used a left join instead of an inner join — that is worth a comment.
