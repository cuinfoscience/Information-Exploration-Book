13  Cleaning Data with SQL

13.1 Learning Objectives

By the end of this chapter, you will be able to:

  • Apply a five-stage Inspect → Clean → Design Schema → Load → Validate workflow to a messy dataset.
  • Write SQL diagnostic queries for missingness, duplicates, inconsistent categories, and invalid ranges.
  • Use SQLite’s built-in string functions (TRIM, UPPER, LOWER, REPLACE) and CASE WHEN expressions to clean columns.
  • Diagnose and repair silent join failures caused by inconsistent key formatting across agency sources.
  • Consolidate cleaned tables from multiple source databases into a single analysis database using DataFrame.to_sql.

13.2 Why SQL for Cleaning

Cleaning data in SQL follows the same logic you learned in pandas — identify missingness, duplicates, inconsistent categories, and invalid ranges — but uses a different set of tools. SQL excels at diagnostic queries: you can characterize problems across an entire table with a single statement, and the declarative syntax makes your logic explicit and auditable. This chapter applies a structured diagnostic framework to three real Colorado government datasets, loads them into a SQLite database, and demonstrates a critical lesson about silent join failures caused by inconsistent key formatting across sources.

The silent join failure in this chapter is the clearest example in the book of a bug that does not look like a bug. A query runs, returns results, and those results are systematically wrong. Reading this chapter carefully will save you days of confusion in your future data work.

13.3 The Diagnostic Framework

Every cleaning session in this chapter follows the same five stages:

Inspect → Clean → Design Schema → Load → Validate

Within the Inspect stage, we run four standard diagnostics, one per category of data quality issue:

  1. Missingness — which columns have nulls, and how many?
  2. Duplicates — are there repeated rows on the expected key columns?
  3. Inconsistent categories — do text columns have variant spellings, extra whitespace, or unexpected values?
  4. Invalid ranges — are numeric and date columns within plausible bounds?

This is the same INSPECT → TRANSFORM → VALIDATE → DOCUMENT workflow from the pandas chapters, specialized for SQL.

13.9 Part 4: The Silent Join Failure

This is the most important section of the chapter. After loading all three tables, we discover that county names are formatted differently in each source:

Table Example county name
deaths Adams County, CO
crime Adams County
population Adams

What happens when you try to join these tables on county name? Nothing matches. The join produces zero rows — silently, with no error message. This is a silent join failure, and it is one of the most dangerous problems in data work because it looks like the query succeeded (it ran without errors) but the result is empty or incomplete.

13.9.1 Demonstrating the failure

pd.read_sql_query("""
    SELECT d.county AS death_county, p.county AS pop_county, COUNT(*) AS n
    FROM deaths d
    JOIN population p ON d.county = p.county
    LIMIT 5;
""", conn)

If this returns zero rows, the county formats do not match.

13.9.2 Diagnosing the mismatch

pd.read_sql_query("""
    SELECT DISTINCT county FROM deaths ORDER BY county LIMIT 10;
""", conn)
pd.read_sql_query("""
    SELECT DISTINCT county FROM population ORDER BY county LIMIT 10;
""", conn)

13.9.3 Fixing the mismatch

The fix is to standardize county names to a common format before loading, or to use SQL string functions to normalize them at query time. The REPLACE() and TRIM() functions in SQLite can help, but the cleanest approach is to fix the data in pandas before loading:

# Standardize all county names to bare names: 'Adams'
# From 'Adams County, CO' → remove ', CO' then remove ' County'
# From 'Adams County' → remove ' County'
# From 'Adams' → already correct

13.9.4 The lesson

County name mismatches across sources are not a quirk of these particular datasets — they are a universal pattern. Any time you join data from different sources on a text key, you must verify that the key values actually match before running the join. The diagnostic pattern is: list the distinct values from each table, compare them, and fix any mismatches before joining.

NotePublic Interest Context: Erosion

The three county name formats — "Adams County, CO", "Adams County", "Adams" — come from three Colorado government agencies that built their data systems independently, at different times, with no shared naming authority. No one decided to make them incompatible. The mismatch is the accumulated result of decades of decentralized infrastructure development.

This is infrastructure erosion: not a sudden break, but a gradual divergence that only becomes visible when you try to connect the systems. The cost is borne by anyone who wants to analyze these datasets together — which is exactly the analysis you would want to perform if you were studying public health outcomes across Colorado counties.

The standardization work you completed (removing “, CO” and ” County”) is a technical fix for a governance problem. The fix lives in your notebook; the problem persists in the source systems. See Chapter 7 for the broader framework.

13.10 SQL Functions for Cleaning

Several SQL functions are useful for cleaning operations:

Function Purpose Example
TRIM(col) Remove leading/trailing whitespace SELECT TRIM(county) FROM deaths;
UPPER(col) Convert to uppercase SELECT UPPER(county) FROM deaths;
LOWER(col) Convert to lowercase SELECT LOWER(county) FROM deaths;
REPLACE(col, old, new) Replace substring SELECT REPLACE(county, ' County', '') FROM deaths;
CAST(col AS type) Change data type SELECT CAST(year AS INTEGER) FROM deaths;
COALESCE(col, default) Replace NULL with default SELECT COALESCE(deaths, 0) FROM deaths;
CASE WHEN Conditional logic See below

13.10.1 CASE WHEN for conditional cleaning

SELECT
    county,
    CASE
        WHEN deaths IS NULL THEN 'Missing'
        WHEN deaths = 0 THEN 'Zero'
        WHEN deaths < 10 THEN 'Suppressed range'
        ELSE 'Normal'
    END AS death_status
FROM deaths;

13.11 Core Diagnostic Patterns

These four query patterns apply to any table in any database:

Missingness: SELECT col, COUNT(*) - COUNT(col) AS n_null FROM table;

Duplicates: SELECT key_cols, COUNT(*) AS n FROM table GROUP BY key_cols HAVING n > 1;

Inconsistent categories: SELECT col, COUNT(*) FROM table GROUP BY col ORDER BY COUNT(*) DESC;

Invalid ranges: SELECT MIN(col), MAX(col), AVG(col) FROM table;

13.12 Saving the Cleaned Tables

After validating each dataset, collect the cleaned, analysis-ready versions into a single SQLite database for use in Chapter 15. That chapter expects one deaths table, one crime table, and one population table — all in colorado_public_health.db, with county names standardized to bare county names (e.g., Adams) and all rows validated.

The cleanest approach is to read each table from its source database into a pandas DataFrame, apply the county-name standardization you diagnosed above, and write to the shared database using to_sql:

import pandas as pd
import sqlite3

# Read cleaned tables from their source databases
conn_deaths = sqlite3.connect('cdc_deaths.db')
conn_crime = sqlite3.connect('cbi_crime.db')
conn_pop = sqlite3.connect('population.db')

deaths = pd.read_sql_query("SELECT * FROM deaths", conn_deaths)
crime = pd.read_sql_query("SELECT * FROM crime", conn_crime)
population = pd.read_sql_query("SELECT * FROM population", conn_pop)

# Standardize county names to bare format (e.g., "Adams County, CO" -> "Adams")
deaths['county'] = (deaths['county']
    .str.replace(r',\s*CO$', '', regex=True)   # remove ", CO"
    .str.replace(r'\s+County$', '', regex=True) # remove " County"
    .str.strip()
)

crime['county'] = (crime['county']
    .str.replace(r'\s+County$', '', regex=True)
    .str.strip()
)
# population already uses bare county names

# Write to shared database
conn_ph = sqlite3.connect('colorado_public_health.db')
deaths.to_sql('deaths', conn_ph, if_exists='replace', index=False)
crime.to_sql('crime', conn_ph, if_exists='replace', index=False)
population.to_sql('population', conn_ph, if_exists='replace', index=False)

print("Tables written to colorado_public_health.db:")
for tbl in ['deaths', 'crime', 'population']:
    n = pd.read_sql_query(f"SELECT COUNT(*) AS n FROM {tbl}", conn_ph).iloc[0, 0]
    print(f"  {tbl}: {n:,} rows")

conn_deaths.close(); conn_crime.close(); conn_pop.close(); conn_ph.close()
TipWhy not write directly from SQL?

SQLite does not have a built-in cross-database INSERT INTO ... SELECT FROM for different .db files. The pandas round-trip (read into DataFrame, standardize, write back) is the practical approach for consolidating multiple source databases into one analysis database.

13.13 Common Issues in This Chapter

WarningA join on a text key returns zero rows

What it means: A silent join failure. Your keys look similar but are not identical — often "Adams" vs "Adams County" vs "Adams County, CO".

How to fix it: Run SELECT DISTINCT county FROM <table> ORDER BY county on each side of the join. Compare the results. Normalize in pandas (or with SQLite TRIM/REPLACE) before joining.

WarningTRIM does not catch some whitespace

What it means: Non-breaking spaces (Unicode U+00A0) and tab characters are whitespace to humans but not to SQLite’s default TRIM. A column can still fail to match after TRIM if the whitespace is unusual.

How to fix it: Check for unusual whitespace in pandas with df['col'].str.contains(r'\s', regex=True) and explicitly replace \xa0 and other characters before exporting. In SQL, REPLACE(col, char(160), ' ') handles non-breaking spaces.

WarningCAST(x AS INTEGER) returns 0 unexpectedly

What it means: SQLite’s CAST returns 0 for strings that cannot be parsed. A column with $1,234 gets stored as text, and CAST silently produces zeros.

How to fix it: Clean currency/comma characters before loading. In SQLite, chain CAST(REPLACE(REPLACE(col, '$', ''), ',', '') AS INTEGER) — but doing the cleanup in pandas first is clearer and easier to audit.

WarningNULL behaves unexpectedly in conditions

What it means: NULL = NULL is NULL (not TRUE), and NULL = 'Adams' is NULL. Any row with NULL in the compared column is silently excluded.

How to fix it: Always use IS NULL and IS NOT NULL for null checks. Use COALESCE(col, default) when you need a default value.

13.14 Key Takeaways

  • Cleaning in SQL follows the same Inspect → Clean → Validate → Document rhythm as pandas; only the syntax changes.
  • The four diagnostic categories — missingness, duplicates, inconsistent categories, invalid ranges — are worth running on every new table.
  • Silent join failures are the most dangerous SQL bug class. Always run SELECT DISTINCT on join keys before joining.
  • SQLite’s type system is permissive. Column names and types in your schema do not enforce what is actually stored — audit after every load.
  • When a technical fix (standardizing county names) encodes a governance problem (three agencies, no shared convention), document the fix in your Known Issues Log and the provenance in your Data Dictionary.

## SQL Functions for Cleaning

Several SQL functions are useful for cleaning operations:

| Function | Purpose | Example |
|----------|---------|---------|
| `TRIM(col)` | Remove leading/trailing whitespace | `SELECT TRIM(county) FROM deaths;` |
| `UPPER(col)` | Convert to uppercase | `SELECT UPPER(county) FROM deaths;` |
| `LOWER(col)` | Convert to lowercase | `SELECT LOWER(county) FROM deaths;` |
| `REPLACE(col, old, new)` | Replace substring | `SELECT REPLACE(county, ' County', '') FROM deaths;` |
| `CAST(col AS type)` | Change data type | `SELECT CAST(year AS INTEGER) FROM deaths;` |
| `COALESCE(col, default)` | Replace NULL with default | `SELECT COALESCE(deaths, 0) FROM deaths;` |
| `CASE WHEN` | Conditional logic | See below |

### `CASE WHEN` for conditional cleaning

```sql
SELECT
    county,
    CASE
        WHEN deaths IS NULL THEN 'Missing'
        WHEN deaths = 0 THEN 'Zero'
        WHEN deaths < 10 THEN 'Suppressed range'
        ELSE 'Normal'
    END AS death_status
FROM deaths;
```

## Core Diagnostic Patterns

These four query patterns apply to any table in any database:

**Missingness**: `SELECT col, COUNT(*) - COUNT(col) AS n_null FROM table;`

**Duplicates**: `SELECT key_cols, COUNT(*) AS n FROM table GROUP BY key_cols HAVING n > 1;`

**Inconsistent categories**: `SELECT col, COUNT(*) FROM table GROUP BY col ORDER BY COUNT(*) DESC;`

**Invalid ranges**: `SELECT MIN(col), MAX(col), AVG(col) FROM table;`

## Saving the Cleaned Tables

After validating each dataset, collect the cleaned, analysis-ready versions into a single SQLite database for use in @sec-sql-reshaping. That chapter expects one `deaths` table, one `crime` table, and one `population` table — all in `colorado_public_health.db`, with county names standardized to bare county names (e.g., `Adams`) and all rows validated.

The cleanest approach is to read each table from its source database into a pandas DataFrame, apply the county-name standardization you diagnosed above, and write to the shared database using `to_sql`:

```{python}
import pandas as pd
import sqlite3

# Read cleaned tables from their source databases
conn_deaths = sqlite3.connect('cdc_deaths.db')
conn_crime = sqlite3.connect('cbi_crime.db')
conn_pop = sqlite3.connect('population.db')

deaths = pd.read_sql_query("SELECT * FROM deaths", conn_deaths)
crime = pd.read_sql_query("SELECT * FROM crime", conn_crime)
population = pd.read_sql_query("SELECT * FROM population", conn_pop)

# Standardize county names to bare format (e.g., "Adams County, CO" -> "Adams")
deaths['county'] = (deaths['county']
    .str.replace(r',\s*CO$', '', regex=True)   # remove ", CO"
    .str.replace(r'\s+County$', '', regex=True) # remove " County"
    .str.strip()
)

crime['county'] = (crime['county']
    .str.replace(r'\s+County$', '', regex=True)
    .str.strip()
)
# population already uses bare county names

# Write to shared database
conn_ph = sqlite3.connect('colorado_public_health.db')
deaths.to_sql('deaths', conn_ph, if_exists='replace', index=False)
crime.to_sql('crime', conn_ph, if_exists='replace', index=False)
population.to_sql('population', conn_ph, if_exists='replace', index=False)

print("Tables written to colorado_public_health.db:")
for tbl in ['deaths', 'crime', 'population']:
    n = pd.read_sql_query(f"SELECT COUNT(*) AS n FROM {tbl}", conn_ph).iloc[0, 0]
    print(f"  {tbl}: {n:,} rows")

conn_deaths.close(); conn_crime.close(); conn_pop.close(); conn_ph.close()
```

::: {.callout-tip}
## Why not write directly from SQL?
SQLite does not have a built-in cross-database `INSERT INTO ... SELECT FROM` for different `.db` files. The pandas round-trip (read into DataFrame, standardize, write back) is the practical approach for consolidating multiple source databases into one analysis database.
:::

## Common Issues in This Chapter

::: {.callout-warning}
## A join on a text key returns zero rows
**What it means:** A silent join failure. Your keys look similar but are not identical — often `"Adams"` vs `"Adams County"` vs `"Adams County, CO"`.

**How to fix it:** Run `SELECT DISTINCT county FROM <table> ORDER BY county` on each side of the join. Compare the results. Normalize in pandas (or with SQLite `TRIM`/`REPLACE`) before joining.
:::

::: {.callout-warning}
## `TRIM` does not catch some whitespace
**What it means:** Non-breaking spaces (Unicode `U+00A0`) and tab characters are whitespace to humans but not to SQLite's default `TRIM`. A column can still fail to match after `TRIM` if the whitespace is unusual.

**How to fix it:** Check for unusual whitespace in pandas with `df['col'].str.contains(r'\s', regex=True)` and explicitly replace `\xa0` and other characters before exporting. In SQL, `REPLACE(col, char(160), ' ')` handles non-breaking spaces.
:::

::: {.callout-warning}
## `CAST(x AS INTEGER)` returns 0 unexpectedly
**What it means:** SQLite's `CAST` returns 0 for strings that cannot be parsed. A column with `$1,234` gets stored as text, and `CAST` silently produces zeros.

**How to fix it:** Clean currency/comma characters before loading. In SQLite, chain `CAST(REPLACE(REPLACE(col, '$', ''), ',', '') AS INTEGER)` — but doing the cleanup in pandas first is clearer and easier to audit.
:::

::: {.callout-warning}
## `NULL` behaves unexpectedly in conditions
**What it means:** `NULL = NULL` is `NULL` (not `TRUE`), and `NULL = 'Adams'` is `NULL`. Any row with `NULL` in the compared column is silently excluded.

**How to fix it:** Always use `IS NULL` and `IS NOT NULL` for null checks. Use `COALESCE(col, default)` when you need a default value.
:::

## Key Takeaways

- Cleaning in SQL follows the same **Inspect → Clean → Validate → Document** rhythm as pandas; only the syntax changes.
- The four diagnostic categories — missingness, duplicates, inconsistent categories, invalid ranges — are worth running on every new table.
- **Silent join failures** are the most dangerous SQL bug class. Always run `SELECT DISTINCT` on join keys before joining.
- SQLite's type system is permissive. Column names and types in your schema do not enforce what is actually stored — audit after every load.
- When a technical fix (standardizing county names) encodes a governance problem (three agencies, no shared convention), document the fix in your Known Issues Log and the provenance in your Data Dictionary.
