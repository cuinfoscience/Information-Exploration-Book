2  Project Organization

A well-organized project is one you can return to after six months and understand immediately. It is also one that someone else — a collaborator, an instructor, a future employer reviewing your portfolio — can open and navigate without asking you questions. This chapter introduces a simple project structure that scales from a homework assignment to a professional analysis. The terminal commands from Chapter 1 — mkdir, cd, ls — are how you create and navigate this structure from the command line.

2.1 Why Structure Matters

When you first start working with data, everything tends to end up in one folder: the raw data files, the cleaned versions, the notebooks, the outputs, and the half-finished experiments. This works until it doesn’t. The moment you have two versions of a dataset and cannot remember which one you cleaned, or you overwrite your raw data with a transformed version, you have a problem that no amount of technical skill can fix.

Good project organization prevents these problems before they arise. It costs almost nothing to set up and saves enormous time later.

2.2 A Minimal Project Structure

Here is a structure that works for the projects in this book and adapts easily to larger work:

my_project/
├── data/
│   ├── raw/           # Original, untouched data files
│   └── clean/         # Cleaned versions ready for analysis
├── notebooks/         # Jupyter notebooks (analysis work)
├── output/            # Tables, figures, exported results
└── README.md          # What this project is about

The key principle is the separation between raw and clean data. Raw data files are sacred — you should never modify them directly. Instead, your notebooks read from raw/, perform transformations, and write results to clean/ or output/. If something goes wrong, you can always start over from the original files.

2.3 Naming Conventions

File names should be descriptive, consistent, and machine-friendly. A few rules that eliminate most naming headaches:

Use lowercase letters, numbers, hyphens, and underscores. Avoid spaces, special characters, and accented letters. Spaces in file names cause problems in the terminal (you have to escape them with backslashes or wrap the name in quotes), and special characters can break scripts that process files automatically.

Include dates in ISO format (YYYY-MM-DD) when versioning matters. report-2026-02-10.csv sorts chronologically; report-Feb-10.csv does not.

Be specific. data.csv tells you nothing. colorado-cannabis-sales-by-county-2014-2024.csv tells you exactly what is inside without opening the file.

2.4 The README File

Every project should have a README.md file in its root directory. This file answers three questions: what is this project about, what data does it use, and how do you run it. Writing README files and keeping a Known Issues Log are part of the broader documentation practices described in Chapter 3. A minimal README for a course project might look like this:

# Cannabis Sales Analysis

Analysis of Colorado cannabis sales by county (2014–2024) for INFO 3401.

## Data

- `data/raw/colorado_cannabis_sales_by_county.csv`: Source file from
  Colorado Department of Revenue. Downloaded 2026-02-10.

## Notebooks

- `notebooks/01-loading.ipynb`: Load and inspect raw data.
- `notebooks/02-cleaning.ipynb`: Clean column names, fix types, remove totals.

Writing this file takes five minutes and makes your project immediately legible to anyone who opens it.

2.5 Working Directories and File Paths in Notebooks

When you open a Jupyter notebook, it runs in the directory where the notebook file lives. Your Python environment — and whether commands like python and jupyter are available from the terminal — must be configured correctly before notebooks will run; see Chapter 4 for setup instructions. If your notebook is in my_project/notebooks/ and your data is in my_project/data/raw/, you need a relative path that goes up one level and then down into data:

df = pd.read_csv('../data/raw/cannabis_sales.csv')

The .. means “go up one directory.” This relative path works on any computer with the same project structure, regardless of where the project folder itself lives. Avoid hardcoding absolute paths like /Users/yourname/Documents/my_project/data/raw/cannabis_sales.csv — they break the moment someone else tries to run your notebook.

TipA common frustration

If pd.read_csv() gives you a FileNotFoundError, the problem is almost always that your notebook’s working directory is not where you think it is. Run import os; print(os.getcwd()) to check, then adjust your path accordingly.

2.6 Version Control (A Brief Preview)

As your projects grow, you will benefit from a version control system like git, which tracks every change you make to every file and lets you revert to any previous state. Git is beyond the scope of this book, but knowing it exists shapes how you organize your work. The project structure described above — with raw data separated from outputs, and a README explaining the project — is designed to work naturally with version control.

2.6.1 .gitignore: Files You Do Not Track

If you use git, some files should never be committed to version control: large binary data files, credentials, automatically generated outputs, and system junk files (like .DS_Store on Mac). You communicate this to git using a .gitignore file — a plain-text list of file patterns to exclude.

A minimal .gitignore for a data analysis project looks like this:

# Data files (too large for version control)
data/raw/
data/clean/*.csv

# Python artifacts
__pycache__/
*.pyc
.ipynb_checkpoints/

# System files
.DS_Store
Thumbs.db

Create this file in your project’s root directory. Git will ignore any file matching these patterns. The practical effect is that your code is version-controlled, but your data files are not — which is usually the right trade-off for course projects.

If you want to explore git, GitHub provides excellent beginner tutorials, and most data science courses cover it in depth. For now, the discipline of keeping raw data untouched and documenting your workflow in notebooks gives you many of the same benefits.
